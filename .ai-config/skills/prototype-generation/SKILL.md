# 原型生成技能

## 目标

在需求接入的第一阶段（Step 3），根据需求的特征**自动判定是否需要生成原型**，若需要则**选择合适的原型形式**（HTML/Figma/低保真草图），帮助需求提出方和开发团队进行视觉化沟通，减少理解偏差。

**触发时机**：Step 2（工作量评估）完成后，进入Step 3前自动判定

**输出**：原型生成决策 + 若需要则输出原型文件

---

## 原型必需性判定规则

### 判定维度

| 维度 | 判定标准 | 分值 |
|---|---|---|
| **功能类型** | 是否涉及UI/交互设计 | 0-3分 |
| **复杂度** | 页面数/功能流程复杂度 | 0-3分 |
| **涉众** | 是否需要与非技术人员沟通 | 0-2分 |
| **验收** | 验收标准中是否涉及UI/流程细节 | 0-2分 |

### 自动判定算法

```
计算总分：功能分 + 复杂分 + 涉众分 + 验收分

总分 ≤ 2     → 不需要原型（纯业务逻辑/后端API）
3-6 分       → 可选原型（简单HTML+示意图）
7-9 分       → 推荐原型（可交互的HTML或Figma）
≥ 10 分      → 必需原型（完整交互原型）
```

### 详细判定标准

#### 维度1：功能类型（0-3分）

| 功能类型 | 示例 | 分值 | 原因 |
|---|---|---|---|
| **纯后端/API** | 数据接口、定时任务、计算引擎 | 0分 | 无UI，不需原型 |
| **库/工具** | 工具类、SDK、算法优化 | 0分 | 无UI，不需原型 |
| **简单表单** | 登录、注册、简单增删改查 | 1分 | UI简单，文字描述足够 |
| **复杂表单** | 多步骤表单、权限配置、高级搜索 | 2分 | 涉及多个交互，需简单示意 |
| **页面流程** | 完整业务流程（订单→支付→物流） | 3分 | 多页面跳转，强烈需要可视化 |
| **数据大屏** | 统计分析、可视化仪表盘 | 3分 | 布局复杂，必需原型 |
| **移动端应用** | 手机应用、响应式设计 | 3分 | 交互密集，必需原型 |

#### 维度2：复杂度（0-3分）

| 复杂度指标 | 0分 | 1分 | 2分 | 3分 |
|---|---|---|---|---|
| **页面数** | 0-1页 | 2-3页 | 4-6页 | 7+页 |
| **功能点** | 1-2个 | 3-5个 | 6-10个 | 11+个 |
| **交互流程** | 单线性 | 简单分支 | 多分支 | 复杂非线性 |
| **状态管理** | 无状态 | 简单状态 | 多状态转移 | 复杂状态机 |

#### 维度3：涉众（0-2分）

| 涉众类型 | 分值 | 说明 |
|---|---|---|
| 仅开发团队内部沟通 | 0分 | 技术人员能理解文字描述 |
| 需要与产品/设计沟通 | 1分 | 非技术人员需要看到可视化 |
| 需要与最终用户沟通 | 2分 | 必须展示给用户看以获得反馈 |

#### 维度4：验收标准（0-2分）

| 验收标准特征 | 分值 | 示例 |
|---|---|---|
| 纯功能性验收（"支持查询"） | 0分 | 无需原型 |
| 包含流程验收（"支持三步审批流"） | 1分 | 需要简单流程图 |
| 包含UI验收（"菜单布局如下"或"需要日期选择器"） | 2分 | 需要原型确认 |

## 原型形式选择

### 三种原型形式对比

| 原型形式 | 工期 | 交互性 | 可视化程度 | 迭代成本 | 适用场景 |
|---|---|---|---|---|---|
| **HTML** | 快（5-10分钟） | 中等 | 高 | 低 | 可交互demo，流程演示 |
| **Figma** | 中等（10-20分钟） | 低 | 很高 | 中等 | 设计稿共享，多方评审 |
| **低保真草图** | 很快（2-5分钟） | 无 | 中 | 很低 | 快速概念验证 |

### 自动选择逻辑

```
判定完"需要原型"后，根据需求特征选择形式：

IF 判定分数 3-4分 THEN
   → 低保真草图（简单线框图）
   理由：只需快速确认概念，成本低

ELSE IF 判定分数 5-7分 THEN
   IF 需求涉及复杂流程 THEN
      → HTML可交互原型
      理由：可以展示页面跳转和数据流
   ELSE
      → Figma设计稿
      理由：静态展示UI布局和样式
      
ELSE IF 判定分数 ≥8分 THEN
   IF 涉众包含最终用户 THEN
      → HTML完整可交互原型
      理由：高保真，可验证用户体验
   ELSE
      → Figma + HTML混合
      理由：Figma做设计规范，HTML做交互demo
```

**输出物**：Markdown中的ASCII线框图或简单描述

│        │ 【用户列表】              │
│        │                          │

**输出物**：可独立打开的HTML文件，包含基本交互

### 3. Figma设计稿

**输出物**：Figma链接或导出图片

**包含内容**：
- 完整的UI设计（颜色、字体、间距）
- 组件库标注
- 交互流程标注
- 响应式设计说明
- 设计规范（颜色、字体、按钮样式等）

**特点**：
- 高保真视觉效果
- 便于设计和开发协作
- 支持多人评审和批注
- 可导出为各种规格

**生成方式**：AI生成Figma设计稿的导出版本（PNG/SVG）或创建链接

## 原型生成触发流程

### 在Step 3中的位置

```
【Step 2完成】工作量评估结果: S/M/L/XL
   ↓
【自动触发】原型必需性判定
   ├─ 分析需求特征（功能类型、复杂度、涉众）
   ├─ 计算总分（0-10分）
   └─ 判定是否需要原型
   
   ├─ 若不需要 → 跳过原型，继续Step 3其他规范
   │
   └─ 若需要 → 选择原型形式
      ├─ 分数3-4: 输出低保真草图
      ├─ 分数5-7: 输出HTML或Figma
      └─ 分数≥8: 输出HTML+Figma混合
      
      ↓
      生成原型文件
      ├─ 放在: docs/prototype/REQ-XXXXXXXX.html 或 .figma
      └─ 需求文档中引入链接
   
   ↓
【输出物规范】更新
   ├─ S/M级不含原型（或可选）
   ├─ L级可能包含原型
   └─ XL级必含原型+设计规范
```

## 原型与工作量的关系

### 原型是否增加工作量评估

```
答案：不增加，但会影响后续工期

理由：
1. 原型只是"视觉化沟通工具"，不是"最终产品"
2. 原型包含的流程和交互，最终都要在代码中实现
3. 但原型可以提前发现需求问题，减少重工

【工作量评估时】
- 评估原型生成的工作量：极小（5-20分钟）
- 这5-20分钟算在"需求澄清"中，不算在代码工期中
- 原型发现的问题如果很大，可能需要重评估

【实际效果】
- 无原型: 20个工作日 + 5个工作日返工 = 25天
- 有原型: 20个工作日 + 1个工作日修改 = 21天
  （原型花10分钟，但节省了5个返工天）
```

## 与AI生成的集成

### HTML原型的自动生成

```javascript
// 伪代码：原型生成算法

function generatePrototype(requirement, prototypeType) {
    
    if (prototypeType === 'HTML') {
        // 1. 解析需求中的页面清单和流程
        const pages = parsePages(requirement);
        const flows = parseFlows(requirement);
        
        // 2. 生成HTML骨架
        const html = generateHTMLStructure(pages);
        
        // 3. 添加交互逻辑
        const interactions = generateInteractions(flows);
        
        // 4. 应用样式（Bootstrap / Ant Design）
        const styled = applyStyles(html, 'ant-design');
        
        // 5. 输出文件
        return outputHTML(styled + interactions);
    }
    
    else if (prototypeType === 'Figma') {
        // 调用Figma API或导出设计稿
        return generateFigmaDesign(requirement);
    }
    
    else if (prototypeType === 'Wireframe') {
        // 生成Markdown线框图
        return generateWireframeMarkdown(requirement);
    }
}
```
