# 工程文档规范化（README / 模块业务说明 / ChangeLog）

## 适用场景（何时调用）

- 完成任意开发任务准备交付（提测、合并、发版）时。
- 用户提出“工程文件规范化”“按规范补齐 README / ChangeLog / 业务文档”“补文档、补日志”等需求时。
- 初始化项目、迭代功能、架构调整、高耦合模块改造结束时。

## 目标（必须满足）

### 1）README.md 规范（强制）

- 项目根目录必须存在 README.md。
- 初始化项目、迭代功能、架构调整时，自动创建/同步更新 README.md。
- README.md 必备核心内容：
  - 项目简介
  - 技术栈
  - 环境依赖
  - 运行方式
  - 目录结构
  - 核心业务

### 2）复杂模块业务文档规范（强制）

- 复杂业务/核心流程/高耦合模块，必须配套独立业务说明文件。
- 文件规则：
  - 路径：docs/modules/
  - 命名：模块名_业务说明.md
- 必备内容：
  - 模块定位
  - 业务流程
  - 依赖关系
  - 数据流向
  - 使用规则
- 业务逻辑变更时，同步更新对应业务说明文件。

### 3）ChangeLog.md 规范（强制）

- 项目根目录必须存在 ChangeLog.md，所有代码变更必须记录。
- 严格格式要求：
  - 最新变更置顶，倒序排列。
  - 同一天的相似/同模块改动自动合并。
  - 分类标签：【新增】【优化】【修复】【重构】【删除】
- 标准格式示例：
  - `## 2026-03-25`
  - `- 【优化】登录模块：优化密码校验逻辑`
  - `- 【修复】订单模块：修复提交参数异常问题`

## 执行步骤（按顺序完成）

### 第 0 步：确定本次“改动范围与归类”

- 汇总本次变更点：新增了什么、修复了什么、优化了什么、重构/删除了什么。
- 归类到 ChangeLog 标签（【新增】【优化】【修复】【重构】【删除】），并给出“模块名：一句话说明”的条目。

### 第 1 步：README.md 检查与同步

1. 若根目录缺失 README.md：创建 README.md。
2. 若已存在：仅在需要时同步更新（初始化、功能迭代、架构调整、运行方式变更、目录结构明显变化等）。
3. README.md 必须包含以下一级标题（可用 `##`）：
   - 项目简介
   - 技术栈
   - 环境依赖
   - 运行方式
   - 目录结构
   - 核心业务

README.md 推荐最小模板（可按项目实际替换内容）：

```markdown
# <项目名称>

## 项目简介
<一句话定位 + 主要场景>

## 技术栈
- <前端/后端/构建工具/运行环境等>

## 环境依赖
- Node.js：<版本要求>
- pnpm / npm：<版本要求>
- <其他关键依赖>

## 运行方式
```bash
<启动命令>
```

## 目录结构
```text
<关键目录与说明>
```

## 核心业务
- <核心业务 1：一句话>
- <核心业务 2：一句话>
```

### 第 2 步：识别“复杂模块”并补齐/更新业务说明

判定建议（命中任意项即视为复杂模块，需 `docs/modules/模块名_业务说明.md`）：

- 涉及核心流程：登录/权限、工单、设备、点检、仓储、生产、质量等主链路。
- 新增或大改：跨多个页面/组件/接口（例如新增视图、路由、权限点、接口联动）。
- 高耦合：同一需求改动触达多个目录（如 `src/views`、`src/api`、`src/store`、`src/router`、权限配置等）。
- 有关键业务规则/状态机/审批流/复杂校验/复杂数据映射。

处理规则：

1. 若本次改动命中复杂模块：在 `docs/modules/` 下创建或更新 `模块名_业务说明.md`。
2. 若仅是小修小改且不涉及业务规则变更：无需新增业务说明文件，但如已有对应文件且本次变更影响其内容，必须同步更新。

业务说明最小模板：

```markdown
# <模块名>：业务说明

## 模块定位
<解决什么问题、面向谁、边界是什么>

## 业务流程
1. <步骤 1>
2. <步骤 2>
3. <步骤 3>

## 依赖关系
- 上游：<依赖的模块/接口/权限/字典等>
- 下游：<影响的模块/数据/报表等>

## 数据流向
- 输入：<页面入参/查询条件/表单字段/路由参数>
- 处理：<关键转换、校验、聚合>
- 输出：<请求参数、落库字段、返回展示>

## 使用规则
- 权限：<按钮/菜单/角色限制>
- 约束：<必填/校验/状态限制/并发限制>
- 异常：<常见错误与处理>
```

### 第 3 步：ChangeLog.md 写入（必须）

1. 若根目录缺失 ChangeLog.md：创建 ChangeLog.md。
2. 获取今天日期（YYYY-MM-DD），将当天条目置顶。
3. 同一天且同模块/同类型的改动合并成更少、更清晰的条目。
4. 每条必须包含分类标签 + 模块名 + 简要说明，避免空泛表述。

ChangeLog.md 推荐结构：

```markdown
## YYYY-MM-DD
- 【修复】<模块名>：<一句话说明>
- 【优化】<模块名>：<一句话说明>

## YYYY-MM-DD
- 【新增】<模块名>：<一句话说明>
```

## 交付检查清单（输出到最终回复里）

- README.md：是否存在；本次是否需要同步更新；已包含必备章节。
- docs/modules：是否识别到复杂模块；已创建/更新哪些 `模块名_业务说明.md`。
- ChangeLog.md：是否存在；是否已记录本次改动；日期置顶、格式与合并规则符合要求。

## 最终输出要求（写在结果里）

- 给出本次写入/更新的文件列表（含路径）。
- 给出 ChangeLog 新增的条目内容（逐条列出）。
- 若判断“不需要新增复杂模块说明”，说明判定理由（例如仅样式/小修复且无业务规则变更）。
