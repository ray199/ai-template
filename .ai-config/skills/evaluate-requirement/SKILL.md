# 技能：需求评估

## 技能描述

对已完成结构化的需求，在进入技术设计之前执行两项评估：
1. **伪需求扫描**：结合项目上下文（代码库、设计文档、历史需求），判断需求是否值得做
2. **输出物推荐**：根据需求复杂度和影响面，推荐需要产出哪些交付物（spec / 迭代计划 / 原型），供人工最终确认

## 触发指令

- `/evaluate` - 对指定需求执行完整评估
- `/evaluate --quick` - 仅做伪需求扫描，跳过输出物推荐
- `/evaluate --artifacts` - 仅做输出物推荐

## 处理流程

```
输入：已 approved 的需求文档（REQ-XXXXXXXX）
       ↓
[Step 1] 加载项目上下文
         - 扫描代码库关键模块
         - 读取 docs/ 下的设计文档和历史需求
         ↓
[Step 2] 执行伪需求扫描（参考 pseudo_checklist.md）
         - 类型A：重复建设检查
         - 类型B：价值存疑检查
         - 类型C：逻辑冲突检查
         - 类型D：技术可行性检查
         ↓
[Step 3] 综合伪需求结论
         - 任一 🔴 → 阻断，必须解决后才能继续
         - 仅 🟡 → 警告，附建议，人工决定是否继续
         - 全 🟢 → 通过
         ↓
[Step 4] 输出物推荐（评估通过后执行）
         - 按复杂度矩阵判断需要哪些输出物
         - 给出推荐理由
         ↓
[Step 5] 输出评估报告，等待人工确认
```

## 输出物推荐规则

### 复杂度判断维度

| 维度 | 低 | 中 | 高 |
|---|---|---|---|
| 涉及页面数 | 0-1 | 2-4 | 5+ |
| 涉及接口数 | 0-2 | 3-6 | 7+ |
| 数据库变更 | 无 | 改字段/加索引 | 新表/结构重构 |
| 影响已有功能 | 无 | 局部影响 | 跨模块/破坏性变更 |
| 工作量预估 | S（1-2天） | M（3-5天） | L/XL（1周+） |

### 推荐矩阵

| 复杂度评级 | 需要 Spec | 需要原型 | 需要迭代计划 |
|---|---|---|---|
| 低（全部低） | ❌ 不需要 | ❌ 不需要 | ❌ 不需要，直接进技术设计 |
| 中（有1项中） | ✅ 建议 | 🔶 视 UI 复杂度 | ❌ 不需要 |
| 高（有1项高） | ✅ 必须 | ✅ 建议 | ✅ 建议 |
| 跨迭代/XL | ✅ 必须 | ✅ 建议 | ✅ 必须 |

> 🔶 = 如果涉及新页面或交互改动，则需要；纯后端逻辑不需要

### 各输出物定义

**Spec（功能规格文档）**
- 用途：明确功能边界、数据结构、接口契约、异常处理
- 生产方式：调用 `project-docs-standards` 技能
- 适用场景：中等以上复杂度，或多人协作开发

**原型（UI 原型）**
- 用途：验证交互设计，避免开发完再返工
- 生产方式：调用 `generate-prototype` 技能
- 适用场景：有新页面、新交互流程，或需要与业务方对齐 UI 预期

**迭代计划**
- 用途：将大需求拆解为可交付的子任务，明确里程碑
- 生产方式：调用 `write-plan` 技能
- 适用场景：工作量 L/XL，或需求横跨多个迭代
