# 伪需求判断标准

> 本文件定义 4 类伪需求的扫描规则和判断标准，供 `evaluate-requirement` 技能使用。
> 每类给出：扫描目标、判断依据、结论输出格式。

---

## 类型 A · 重复建设

**定义**：系统中已存在相同或高度相似的功能，本次需求属于重复投入。

### 扫描目标
- 代码仓库：搜索相关模块、接口、组件名称
- 历史需求文档：检索关键词相似的已完成需求
- 设计文档/spec：查找相同业务场景的已有方案

### 判断标准

| 相似程度 | 结论 |
|---|---|
| 功能完全一致，仅入口/展示不同 | 🔴 高风险伪需求，建议复用或扩展现有功能 |
| 核心逻辑 70% 重叠，有差异点 | 🟡 疑似重复，建议明确差异点后决定是新建还是扩展 |
| 业务场景相似但实现完全不同 | 🟢 非重复，可继续 |

### 输出格式
```
[类型A · 重复建设]
结论：🔴 / 🟡 / 🟢
发现：在 [文件/文档位置] 中存在 [描述]，与本需求 [相似点] 高度重叠
建议：[具体建议]
```

## 类型 B · 价值存疑（ROI 不划算）

**定义**：需求的业务价值模糊或可量化收益极低，投入产出比不合理。

### 判断标准

逐项检查以下问题，有 2 项以上"否"则标记风险：

| 检查项 | 判断 |
|---|---|
| 需求文档中有明确可量化的目标（数字/状态）？ | 是 / 否 |
| 有明确的受益用户群体和使用频率估算？ | 是 / 否 |
| 优先级定为 P0/P1 有明确的业务依据？ | 是 / 否 |
| 验收标准是否能直接验证业务目标达成？ | 是 / 否 |

### 输出格式
```
[类型B · 价值存疑]
结论：🔴 / 🟡 / 🟢
发现：以下信息缺失或不充分：[列出具体问题]
建议：在推进前，请提需方补充 [具体信息]，否则建议降低优先级至 P2 或暂缓
```

## 类型 C · 逻辑冲突

**定义**：需求与现有功能、设计原则、数据模型或业务逻辑存在矛盾。

### 扫描目标
- 设计文档/spec：检查是否有相反的设计决策记录
- 历史需求文档：检查是否有明确拒绝过类似方案的记录
- 数据库结构（若可访问）：检查数据模型约束是否与需求矛盾

### 判断标准

| 冲突类型 | 严重程度 |
|---|---|
| 与已有业务规则直接矛盾（如权限、状态机） | 🔴 必须解决才能继续 |
| 与现有 UI/UX 设计原则不一致 | 🟡 需明确是修改原有还是例外处理 |
| 与历史需求的决策方向相反，但无明确冲突 | 🟡 需说明变更理由 |
| 无冲突 | 🟢 可继续 |

### 输出格式
```
[类型C · 逻辑冲突]
结论：🔴 / 🟡 / 🟢
发现：本需求与 [来源文档/模块] 中的 [具体内容] 存在冲突，具体表现为 [描述]
建议：[选项A：调整需求 / 选项B：更新现有设计 / 选项C：明确例外处理方式]
```

---

## 类型 D · 技术可行性风险

**定义**：需求在当前技术栈、架构或资源条件下实现代价极高，或存在根本性障碍。

### 判断维度

| 维度 | 扫描方式 |
|---|---|
| 架构兼容性 | 对照 `01_tech_stack.mdc` 和现有代码结构 |
| 性能合理性 | 判断数据量/频率是否在现有架构可承受范围 |
| 外部依赖 | 是否依赖不受控的第三方服务或未引入的技术组件 |
| 工作量合理性 | 预估复杂度，判断是否需要拆分 |

### 输出格式
```
[类型D · 技术可行性风险]
结论：🔴 / 🟡 / 🟢
发现：[具体风险描述]
预估影响：工作量 [S/M/L/XL]，需要引入 [新技术/依赖]（是/否）
建议：[可行方案 / 替代方案 / 建议拆分为：子需求1、子需求2]
```
