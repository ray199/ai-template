# 技能：测试与验证

## 技能描述

对已完成编码的需求，驱动 QA 代理执行系统化测试。  
聚焦三件事：**测试用例设计**（覆盖业务场景）、**问题记录**（分级管理缺陷）、**验收结论**（给出明确的通过/阻断结论）。

## 触发指令

- `/test` - 启动完整测试流程（需提供 need_id）
- `/test --cases` - 仅输出测试用例，不执行
- `/test --regression` - 仅执行回归测试（验证已有功能未受影响）
- `/test --report` - 汇总当前测试进度，输出阶段性报告

## 处理流程

```
输入：已确认的需求文档 + 技术设计文档 + 编码完成报告
       ↓
[Step 1] 读取需求验收标准（acceptance 字段）
         作为测试用例的直接来源
         ↓
[Step 2] 生成测试用例集
         - 正常路径（Happy Path）
         - 边界值和异常场景
         - 权限和安全场景
         - 回归场景（已有功能）
         ↓
[Step 3] 执行测试（或标注人工测试点）
         - AI 可执行：接口测试（curl / Postman 用例生成）
         - 需人工执行：UI 交互、端到端流程
         ↓
[Step 4] 记录问题，按级别分类
         - P0 阻断：需立即修复，不修复不验收
         - P1 严重：需在上线前修复
         - P2 一般：可计划修复，不阻断上线
         - P3 建议：优化类，可选择性处理
         ↓
[Step 5] 输出测试报告，给出验收结论
```

## 测试用例设计规则

### 用例从验收标准推导

需求文档中每条 `acceptance` 验收标准，必须对应至少 1 个正常用例 + 1 个异常用例：

```
验收标准：当用户提交申请时，系统应在 3 秒内返回申请编号
→ 正常用例：提交合法申请，验证返回时间 < 3s，且包含申请编号
→ 异常用例：提交缺少必填字段的申请，验证返回 400 + 错误码 XXX_001
```

### 必须覆盖的通用场景

| 场景类型 | 说明 | 优先级 |
|---|---|---|
| 参数为空 / null | 必填字段缺失 | P0 |
| 参数越界 | 字符串超长、数字超范围 | P0 |
| 重复提交 | 相同请求连续提交 2 次 | P1 |
| 无权限访问 | 未登录 / 无对应权限角色访问 | P0 |
| 并发场景 | 关键操作（扣减、抢占）并发提交 | P1（如涉及） |
| 数据不存在 | 传入不存在的 ID | P0 |
| 软删除数据 | 对已删除数据执行操作 | P1 |

1. [步骤描述]

**请求示例**（接口测试）
```bash
curl -X POST http://localhost:8080/api/v1/xxx \

  -H "Content-Type: application/json" \

    "field1": "value1"
  }'
```

**预期结果**
- HTTP 状态码：200

**实际结果**：[测试执行后填写]

**状态**：⏳ 待执行 / ✅ 通过 / ❌ 失败

## 回归测试范围判断规则

| 变更类型 | 回归范围 |
|---|---|
| 仅新增接口和新表 | 最小回归：验证新功能不影响已有接口的健康检查 |
| 修改已有 Service 方法 | 中等回归：测试该 Service 的所有调用方 |
| 修改数据库表结构（加字段） | 中等回归：验证原有数据读写不受影响 |
| 修改公共组件 / 工具类 | 全量回归：覆盖所有依赖该组件的模块 |
| 破坏性变更（接口 / 表结构重构） | 全量回归 + 手动端到端验证 |
