# 变更跟踪技能

## 目标

在需求生命周期中，自动检测和追踪需求变更，按影响面分级，对小变更自动生成闭环的变更记录，对大变更引导通过标准流程重新评估。

**触发时机**：需求已 approved 后，收到变更请求时触发

**输出**：变更等级 + 对应的处理路径

---

## 变更等级分类（点线面体）

### 分级标准

| 等级 | 名称 | 影响范围 | 影响度 | 应对方式 |
|---|---|---|---|---|
| **点** | Point | 单一字段或细节修改 | <5% | 自动生成变更记录，状态驱动 |
| **线** | Line | 单个功能点或模块内修改 | 5-20% | 生成变更记录，需人工确认 |
| **面** | Surface | 多个功能点或跨模块修改 | 20-50% | 生成变更记录，引导重新评估 |
| **体** | Volume | 核心业务逻辑或架构变更 | >50% | 建议重新走完整接入流程 |

### 判定逻辑

```
检查变更内容
   ↓
1. 数字化评分：影响的字段数 + 技术复杂度 + 业务范围
   
2. 点(Point)判定
   - 仅修改 1-2 个字段（如截止日期、优先级、非关键字段）
   - 无需改动已有代码逻辑
   - 无DB变更
   → 是 → 等级=点

3. 线(Line)判定（若非点）
   - 修改 3-5 个字段或单个功能点
   - 或需要修改 1-2 个现有模块内的逻辑
   - DB 变更控制在1张表内
   → 是 → 等级=线

4. 面(Surface)判定（若非点/线）
   - 修改 6+ 个字段或多个功能点
   - 或涉及 2-3 个跨模块的逻辑修改
   - DB 变更跨越多张表或新增表
   → 是 → 等级=面

5. 体(Volume)判定（若非点/线/面）
   - 覆盖需求的核心业务逻辑 (goal/acceptance 本质改变)
   - 或涉及 3+ 个模块或架构变更
   - 或预期增加工作量 >50%
   → 是 → 等级=体
```

---

## 各等级的处理策略

### 点(Point)：自动闭环

**触发条件**：仅修改非关键字段

**处理流程**
```
检测到点变更
   ↓ 自动生成
变更记录（ChangeLog）
   ├─ 字段：from/to 值
   ├─ 原因：用户提供的变更说明
   ├─ 时间戳：变更时间
   └─ 状态：pending → completed（在阶段4验收时自动标记）
   ↓
更新原需求文档
   ├─ 版本号递增（v1.0 → v1.1）
   ├─ 修改字段生效
   └─ 保留变更历史记录
   ↓
自动输出
   └─ 变更已记录，将在交付验收时完成确认
```

**输出格式**
```markdown
## ✅ 点变更 - 已自动记录

- **变更 ID**: CHANGE-XXXXXXXX-001
- **变更类型**: Point（单字段修改）
- **影响度**: <5%
- **涉及字段**: deadline（2024-03-31 → 2024-04-15）

### 变更记录
| 字段 | 原值 | 新值 | 原因 |
|---|---|---|---|
| deadline | 2024-03-31 | 2024-04-15 | 产品方申请后延2周 |

### 状态生命周期
- 当前：pending（待在阶段4验收时完成）
- 验收时：自动标记为 completed
- 备注：变更不影响已有代码/设计，无需重新评估

---
**下一步**: 继续开发，变更将在交付验收时自动确认
```

### 线(Line)：推荐确认

**触发条件**：单个功能点或模块内修改，影响 5-20%

**处理流程**
```
检测到线变更
   ↓
提示用户确认变更的必要性
   ├─ 影响评估：这个变更会影响 [X] 个相关字段
   ├─ 工作量评估：预期增加工作量 [估算]
   └─ 选项 A: 确认变更 / B: 分离为子需求
   ↓
用户选择
   │
   ├─ 选A: 确认变更
   │   ↓
   │   生成变更记录
   │   ├─ 状态：需确认
   │   ├─ 影响范围：列出相关模块/字段
   │   └─ 工作量变化：从 M → L（举例）
   │   ↓
   │   更新需求文档 + 评估报告
   │   └─ 版本号递增，标注"变更已重新评估"
   │
   └─ 选B: 分离为子需求
       ↓
       拆分为新需求
       └─ 保留原需求不变，新需求单独接入和评估
```

**输出格式**
```markdown
## ⚠️ 线变更 - 需要确认

- **变更 ID**: CHANGE-XXXXXXXX-002
- **变更类型**: Line（功能点修改）
- **影响度**: 12%
- **原工作量**: M（3-5天）
- **预期工作量**: L（1-2周）

### 变更内容
| 维度 | 原值 | 新值 | 影响评估 |
|---|---|---|---|
| 目标 | 支持基础导出 | 支持自定义筛选后导出 | 需扩展查询逻辑 |
| 验收标准 | +1 条 | +2 条 | 新增2个验收场景 |
| 影响模块 | product | product, order | 新增订单模块依赖 |
| DB 变更 | 无 | 新增1个索引 | 优化筛选查询性能 |

### 推荐处理方案
**方案 A: 确认变更（推荐）**
- 预期额外工作量：3-4天
- 风险：需对接订单模块接口，建议提前沟通

**方案 B: 分离为子需求**
- 原需求保持不变，先交付基础导出
- 自定义筛选单独作为后续优化需求

---
**请选择**: 回复 `确认变更` 或 `分离为子需求`
```

### 面(Surface)：重新评估

**触发条件**：多个功能点或跨模块修改，影响 20-50%

**处理流程**
```
检测到面变更
   ↓
强烈建议重新走完整评估流程
   ├─ 原因：变更影响 > 20%，原评估基础已改变
   ├─ 影响：涉及 3+ 个相关字段或 2+ 个模块
   └─ 风险：工作量可能翻倍，需重新评估架构
   ↓
用户选择
   │
   ├─ 选A: 同意重新评估
   │   ↓
   │   更新需求文档（保存原版本）
   │   ↓
   │   触发 `/evaluate` 命令
   │   └─ 重新执行工作量联合评估
   │   ↓
   │   生成新的评估报告
   │   └─ 对标原评估，展示影响差异
   │
   └─ 选B: 保持原评估，接受风险
       ↓
       生成变更记录 + 风险声明
       └─ 标注"已知风险：工作量可能超出原评估"
```

**输出格式**
```markdown
## 🔴 面变更 - 建议重新评估

- **变更 ID**: CHANGE-XXXXXXXX-003
- **变更类型**: Surface（多功能点/跨模块修改）
- **影响度**: 35%

### 变更内容摘要
- 原目标：支持用户权限配置
- 新目标：支持用户权限配置 + 部门权限 + 审批流
- 新增字段：5 个，涉及 3 个模块（user, dept, approval）
- 预期工作量倍数：1.5-2x

### 风险评估
🔴 **高风险项**：
1. 新增部门权限与现有订单关联逻辑冲突（需设计协调）
2. 审批流与现有工作流集成复杂度未知
3. 权限缓存策略需重新设计（性能风险）

### 推荐处理方案

**方案 A: 重新完整评估（强烈推荐）** ⭐
```
1. 保存原版本 (REQ-XXX-v1.0)
2. 更新需求文档为新版本 (REQ-XXX-v2.0)
3. 执行完整评估：/evaluate REQ-XXX
4. 团队确认新的工作量和风险
5. 签字生效
```
预期时间：2-4 小时

**方案 B: 拆分为多个子需求**
```
- 优先级1: 用户权限配置 (原需求，保持不变)
- 优先级2: 部门权限配置 (新子需求)
- 优先级3: 审批流集成 (新子需求)
```

**方案 C: 保持原评估，接受风险**
```
⚠️ 警告：已知以下风险
- 工作量可能超出原评估 50-100%
- 可能导致项目 deadline 延期
- 需后期重新分配资源
```

---
**请选择**: 回复 `执行重新评估` 或 `方案B拆分` 或 `方案C接受风险`
```

### 体(Volume)：拒绝并建议重新接入

**触发条件**：核心业务逻辑改变或架构变更，影响 >50%

**处理流程**
```
检测到体变更
   ↓
系统自动拒绝此变更形式
   ├─ 原因：需求本质改变，超出变更范围
   ├─ 建议：请重新作为新需求接入
   └─ 说明：这个变更的工作量可能接近甚至超过原需求
   ↓
推荐处理方式
   ├─ 选项 A: 作为新需求重新接入评估
   └─ 选项 B: 拆分为"原需求 + N个新子需求"
```

**输出格式**
```markdown
## ❌ 体变更 - 建议重新接入

- **变更 ID**: CHANGE-XXXXXXXX-004
- **变更类型**: Volume（核心业务逻辑改变）
- **影响度**: >50%（无法继续按原方案进行）

### 检测内容
| 维度 | 原需求 | 变更后 | 评估 |
|---|---|---|---|
| 核心目标 | 用户权限管理 | 用户/部门/岗位/数据行四维权限系统 | ✗ 本质改变 |
| 业务逻辑 | 基于 role 的权限 | 基于 RBAC + ABAC 的多维权限 | ✗ 架构重构 |
| 预期工作量 | M (3-5天) | XL (需拆分) | ✗ 工作量翻倍 |
| 风险等级 | 中 | 高 | ✗ 系统架构风险 |

### 为什么不能继续作为变更

这个改动的工作量已经接近甚至超过原需求本身，继续按"变更"处理会：
- ❌ 导致工作量评估严重偏离实际
- ❌ 增加项目风险，可能导致延期
- ❌ 无法明确责任边界和交付标准

### 建议处理方案

**方案 A: 拆分为"原需求 + 新需求"（推荐）** ⭐
```
┌─ REQ-XXX (原): 用户权限管理 (M等级，继续执行)
└─ REQ-YYY (新): 四维权限系统重构 (XL等级，后续规划)
```
时间表：
- REQ-XXX: 按原计划继续开发
- REQ-YYY: 单独接入、评估、排期

**方案 B: 放弃原需求，重新规划整体权限系统**
```
撤销原需求，作为新需求重新完整规划权限系统
预期：重新接入评估、重新排期、重新评审
```

---
**请选择**: 回复 `方案A拆分` 或 `方案B重新规划`
```

---

## 完成标准

变更跟踪功能完成标准：

- [ ] 自动识别变更等级（点/线/面/体）准确
- [ ] 生成了对应的处理策略和推荐方案
- [ ] 变更记录格式清晰，包含影响评估
- [ ] 不同等级的用户交互适当（点自动、线推荐、面重评、体拒绝）
- [ ] 变更历史可追踪（版本号递增、保留旧版本）
