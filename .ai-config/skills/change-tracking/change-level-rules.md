# 变更等级判定规则

## 目标

提供明确的量化标准，自动判定需求变更属于点/线/面/体中的哪一个等级，从而选择对应的处理策略。

---

## 判定维度

### 维度1：变更涉及的字段数量

**需求文档的核心字段清单**
```
优先级字段（必填）：
  - goal：需求目标
  - acceptance：验收标准
  - deadline：截止日期

重要字段（业务复杂度相关）：
  - background：背景/原因
  - scope.in：需要做的事
  - scope.out：不需要做的事
  - stakeholders：干系人/影响方

技术字段：
  - tech_constraints.技术栈：技术约束
  - tech_constraints.依赖系统：外部依赖
  - related_requirements：相关需求
  - non_functional：非功能需求（性能、安全、可用性）

其他字段：
  - priority：业务优先级
  - 评估结果：工作量等级、工期评估
```

**字段分类和权重**
| 字段分类 | 字段列表 | 修改影响度 | 备注 |
|---|---|---|---|
| 关键字段（权重×3） | goal, acceptance, scope.in/out | 高 | 修改意味着需求本质改变 |
| 重要字段（权重×2） | background, stakeholders, non_functional | 中 | 修改影响工作量 |
| 约束字段（权重×2） | tech_constraints, related_requirements | 中 | 修改影响技术方案 |
| 普通字段（权重×1） | priority, deadline, 其他 | 低 | 修改不影响交付物 |

### 维度2：对已有代码/设计的影响

| 影响类型 | 示例 | 影响分数 |
|---|---|---|
| 无需改动 | 仅改 deadline、priority | 0 分 |
| 仅需修改已有模块内逻辑 | 在现有 Service 中追加 1-2 个方法 | 1 分 |
| 需修改多个已有模块 | 修改 2-3 个模块的交互逻辑 | 2 分 |
| 需引入新模块/组件 | 新增 1 个新的 Service/Controller | 2 分 |
| 架构重构 | 跨模块重构，影响数据流 | 3 分 |

### 维度3：数据库变更复杂度

| DB 变更类型 | 示例 | 影响分数 |
|---|---|---|
| 无 DB 变更 | 仅改业务逻辑，无表结构变化 | 0 分 |
| 字段级修改 | 某表新增 1-2 个字段 | 1 分 |
| 表级修改 | 新增 1 张表或主要索引变更 | 2 分 |
| 跨表修改 | 涉及 2+ 张表的变更 | 3 分 |
| 数据迁移 | 涉及数据重构、历史数据迁移 | 4 分 |

### 维度4：业务范围扩展

| 范围 | 示例 | 影响分数 |
|---|---|---|
| 单一细节调整 | 改字段名、改验证规则 | 0 分 |
| 单个功能点扩展 | 在已有功能上加选项 | 1 分 |
| 多个功能点扩展 | 新增 2-3 个关键功能 | 2 分 |
| 跨模块新功能 | 涉及多个模块的新流程 | 3 分 |
| 业务逻辑重构 | goal/acceptance 本质改变 | 4 分 |

## 自动判定算法

### 第一步：计算总影响分数

```
总分 = 字段影响分数 + 代码影响分数 + DB影响分数 + 业务范围分数
```

**字段影响分数计算**
```
对每个被修改的字段，按权重计算：
- 关键字段修改：+3 分/个
- 重要字段修改：+2 分/个
- 约束字段修改：+2 分/个
- 普通字段修改：+1 分/个

字段影响分数 = 各字段分数的和
```

### 第二步：映射到等级

```
总分 ≤ 2       → 点 (Point)
3-8 分         → 线 (Line)
9-16 分        → 面 (Surface)
≥ 17 分        → 体 (Volume)

特殊规则（覆盖总分）：
- 若任意关键字段修改（goal/acceptance 本质改变）→ 直接升为 体
- 若涉及架构重构或业务逻辑重构分数 ≥ 3 → 直接升为 面/体
```

### 第三步：交叉验证

```
若计算结果为 A 等级，进行交叉验证：

IF 代码影响分数 + DB影响分数 ≥ 4 AND 业务范围分数 ≥ 2 THEN
   升级为更高等级 (点→线, 线→面, 面→体)

IF 关键字段中有任何修改 AND 影响 >1 个其他字段 THEN
   升级为体（因为目标本身改变）
```

## 实际判定示例

### 示例1：修改 deadline（点）

**变更内容**：deadline 从 2024-03-31 改为 2024-04-15

**计分**
```
字段影响：deadline 是普通字段 → +1 分
代码影响：无需改动代码 → +0 分
DB影响：无DB变更 → +0 分
业务范围：无业务变更 → +0 分

总分 = 1 分 → 点 (Point)
```

**判断**：✅ 等级确认为点

---

### 示例2：扩展导出功能支持筛选（线）

**变更内容**：
- scope.in：从"支持全量导出"改为"支持筛选后导出"
- 新增 1 个 acceptance：筛选导出场景
- 新增依赖模块：order 模块

**计分**
```
字段影响：
  - scope.in 修改（重要字段）→ +2 分
  - acceptance 新增 1 条 → +1 分
  - tech_constraints.依赖系统 增加 → +2 分
  小计：+5 分

代码影响：
  - 需在现有 ProductService 中追加查询逻辑 → +1 分
  - 需与 OrderService 新建依赖 → +1 分
  小计：+2 分

DB影响：
  - 新增 1 个查询索引（product_id + order_id） → +1 分
  小计：+1 分

业务范围：
  - 单个功能扩展（导出的形式） → +1 分
  小计：+1 分

总分 = 5 + 2 + 1 + 1 = 9 分 → 面 (Surface)

交叉验证：
  - 代码 + DB = 2 + 1 = 3，业务范围 = 1 → 不升级
  - 关键字段是否改变？否 → 不升级

最终判级：面 (Surface)
```

**判断**：⚠️ 等级为面，建议重新评估

---

### 示例3：从用户权限改为四维权限系统（体）

**变更内容**：
- goal：从"支持用户权限配置"改为"建立四维权限系统（用户/部门/岗位/数据行）"
- acceptance：从 2 条改为 8 条（新增多维权限验证场景）
- scope：新增部门、岗位、数据行权限管理
- 影响模块：user, dept, position, data_access（4 个模块）
- DB：新增 4 张表，修改 2 张原表

**计分**
```
字段影响：
  - goal 本质改变（关键字段）→ +3 分
  - acceptance 大幅扩展（重要字段）→ +2 分
  - scope.in 大幅扩展（重要字段）→ +2 分
  - tech_constraints 需重新定义（约束字段）→ +2 分
  小计：+9 分

代码影响：
  - 架构重构，跨模块重设计 → +3 分
  小计：+3 分

DB影响：
  - 新增 4 张表 + 修改 2 张原表 → +4 分
  小计：+4 分

业务范围：
  - 业务逻辑重构（从 RBAC 到 RBAC+ABAC） → +4 分
  小计：+4 分

总分 = 9 + 3 + 4 + 4 = 20 分 → 体 (Volume)

交叉验证：
  - 关键字段修改（goal）+ 影响 > 1 个其他字段 → 必为体 ✓

最终判级：体 (Volume)
```

**判断**：❌ 等级为体，建议重新接入

---

### 示例4：修改验收标准（边界情况）

**变更内容**：
- 修改 acceptance 的 1 条验收标准的细节
- 不影响技术方案，无需改代码

**计分**
```
字段影响：
  - acceptance 修改（重要字段） → +2 分
  
代码影响：无 → +0 分
DB影响：无 → +0 分
业务范围：细节调整 → +0 分

总分 = 2 分 → 线 (Line)

但交叉验证：
  - 是否涉及关键字段本质改变？
    否，只是验证标准的细节措辞修改
  - 业务逻辑是否改变？
    否，仍是相同的功能
  
降级规则：若仅是表述修改或细微澄清，可降为点
```

**判断**：⚠️ 等级为线，但若仅是表述澄清可认定为点

## 变更检测的自动化步骤

### Step 1：收集变更信息

```
1. 读取原需求文档（version v1.0）
2. 读取新提交的变更内容（version v1.1 draft）
3. 逐字段对比，列出所有修改字段
4. 记录变更原因和业务背景
```

### Step 2：字段差异分析

```
对每个修改字段：
1. 判定属于哪一类字段（关键/重要/约束/普通）
2. 判定修改的方向（新增字段值/修改字段值/删除字段值）
3. 计算该字段的影响分数

汇总所有字段的影响分数
```

### Step 3：代码影响评估

```
对于 scope 修改：
1. 提取原 scope.in 的关键词
2. 提取新 scope.in 的关键词
3. 对比新增的关键词，推断涉及的模块
4. 查找代码库中相关模块，判定修改范围

评分方式：
- 无需改动 → 0 分
- 单模块内追加 → 1 分
- 多模块交互 → 2 分
- 新模块 → 2 分
- 架构重构 → 3 分
```

### Step 4：DB 影响评估

```
对于 scope 或 tech_constraints 修改：
1. 查找已有的 DB 设计文档
2. 预判新功能的 DB 需求
3. 对比原 DB 设计，判定变更范围

评分方式：
- 无变更 → 0 分
- 字段级 → 1 分
- 表级 → 2 分
- 跨表 → 3 分
- 数据迁移 → 4 分
```

### Step 5：总分计算与等级判定

```
总分 = 字段影响 + 代码影响 + DB影响 + 业务范围

等级映射：
  ≤2 → Point
  3-8 → Line
  9-16 → Surface
  ≥17 → Volume

特殊规则检查（可升级或拒绝）
```
