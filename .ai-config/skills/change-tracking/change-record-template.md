# 变更记录模板

## 目标

定义变更记录（ChangeLog）的标准格式，确保变更信息完整、可追踪、便于汇总验收。

---

## 标准变更记录格式

### 文件位置

```
docs/requirements/backlog/
├─ REQ-XXXXXXXX.md (原需求文档)
├─ REQ-XXXXXXXX-changelog.md (变更历史，多个变更汇聚)
└─ REQ-XXXXXXXX-v1.0-backup.md (版本备份)
```

### 变更历史文件结构

**文件名**: `REQ-XXXXXXXX-changelog.md`

```markdown
# 需求变更历史 - REQ-XXXXXXXX

## 需求信息
- **need_id**: REQ-XXXXXXXX
- **title**: [需求标题]
- **当前版本**: v1.3
- **首次创建**: 2024-03-10
- **最后更新**: 2024-03-18

---

对于**点(Point)等级的变更**，可以使用简化格式（自动生成，无需人工编写详细内容）：

### 点变更简化模板

```markdown
## 点变更记录

### CHANGE-XXXXXXXX-001
**等级**: Point | **日期**: YYYY-MM-DD

| 字段 | 原值 | 新值 | 原因 |
|---|---|---|---|
| deadline | 2024-03-31 | 2024-04-15 | 产品申请后延 |

**状态**: pending (阶段4自动完成) ✓
```

## 线/面变更的详细记录格式

对于**线(Line)和面(Surface)等级的变更**，使用完整格式：

### 线/面变更详细模板

```markdown
## 线变更记录

### CHANGE-XXXXXXXX-002
**等级**: Line | **日期**: YYYY-MM-DD | **优先级**: 推荐确认

### 变更内容
[详细描述变更的内容和原因]

### 影响评估

**涉及字段**:
| 字段 | 原值 | 新值 | 权重 | 影响 |
|---|---|---|---|---|
| scope.in | A | B | 重要 | +2分 |
| acceptance | X条 | Y条 | 重要 | +1分 |

**代码影响**:
- [具体影响的模块和逻辑]

**DB影响**:
- [具体DB变更]

**工作量变化**:
- 原: M (3-5天)
- 新: M+ (预期+1-2天)

### 推荐方案

**方案A: 确认变更** (推荐)
- 预期额外工作量: [X天]
- 风险等级: [中/高]
- 依赖协调: [列出需要协调的事项]

**方案B: 分离为子需求**
- 原需求保持
- 新功能单独作为后续优化

### 用户选择
**选择**: ☐ 确认 ☐ 分离

### 状态流转
needs_confirm → [用户选择] → approved/separated
```

## 体变更的记录格式

对于**体(Volume)等级的变更**，直接拒绝并建议处理方案：

### 体变更拒绝模板

```markdown
## 体变更记录

### CHANGE-XXXXXXXX-003
**等级**: Volume (核心业务改变) | **日期**: YYYY-MM-DD | **状态**: ❌ 拒绝

### 变更内容
[描述申请的变更]

### 拒绝原因
此变更涉及以下方面：
- ❌ 需求goal本质改变
- ❌ 影响 3+ 个模块
- ❌ 工作量预期倍增
- ❌ 架构重构必需

**结论**: 不符合"变更"的定义，建议重新规划

### 推荐处理方案

**方案A: 拆分为"原需求 + 新需求"** (推荐)
```
原需求保持: REQ-XXX (M, 继续开发)
新子需求1: REQ-YYY (待评估, 后续规划)
新子需求2: REQ-ZZZ (待评估, 后续规划)
```

**方案B: 撤销原需求，重新规划整体方案**
```
重新接入、评估、排期
```

### 用户选择
**选择**: ☐ 方案A拆分 ☐ 方案B重新规划

### 状态流转
rejected → [用户选择] → separated/new_intake
```
