# Step 0A：项目上下文感知规则

## 目标

自动扫描项目目录，识别项目类型（新/旧），加载可利用的资源清单，为后续的需求结构化提供上下文支持。

**触发时机**：每次 `/intake` 触发时自动执行（无需人工干预）

**执行时间**：5-10秒（快速扫描）

**输出**：项目上下文摘要 + 可利用资源清单

---

## 扫描流程

### 步骤1：识别规范成熟度

按以下优先级识别项目拥有的规范文档：

| 级别 | 文件存在 | 说明 | 可用资源 |
|---|---|---|---|
| **完整** | `claude.md` 存在 | 有完整的AI编程规范 | 加载所有规范文件 + .ai-config/ |
| **标准** | `.ai-config/` 存在 | 有AI工具配置 | 加载 rules/ + skills/ 下的规范 |
| **基础** | `README.md` 存在 | 仅有项目说明 | 加载基本信息 |
| **空白** | 上述都不存在 | 新项目，无规范 | 提示建议初始化 |

### 步骤2：识别项目阶段

根据代码库和文档情况判断项目是新建还是迭代：

| 特征 | 判断 | 处理路径 |
|---|---|---|
| `docs/requirements/` 有内容 | 旧项目-有需求历史 | 加载历史需求作参考 |
| `src/` 或 `app/` 有代码 | 旧项目-迭代开发 | 加载已有模块清单 |
| 只有配置文件，无docs/和src/ | 新项目 | 提示建议初始化规范 |
| 仓库为空或刚初始化 | 新项目 | 提示完整初始化步骤 |

### 步骤3：加载可用资源

按优先级加载以下资源：

| 资源类型 | 文件路径 | 用途 | 加载数量 |
|---|---|---|---|
| **技术栈** | `.ai-config/rules/01_tech_stack.mdc` | 自动填写 tech_constraints | 完整 |
| | `README.md` 的技术栈部分 | 补充信息 | 简要 |
| **历史需求** | `docs/requirements/done/` | 识别相似功能 | 最近5条 |
| | `docs/requirements/backlog/` | 了解进行中的需求 | 全部 |
| **已有模块** | 扫描 `src/main/java/` 或对应目录 | 判断影响范围 | 一级目录 |
| **设计文档** | `docs/design/` | 参考已有设计 | 最近3条 |
| **迭代计划** | `docs/requirements/backlog/` 下的 `*-plan.md` | 了解项目规划 | 全部 |

## 自动补全的字段

基于Step 0A的上下文，以下字段可以在Step 1时自动填写：

### 可以自动填写的字段（确定性高）

| 字段 | 数据来源 | 标注方式 | 示例 |
|---|---|---|---|
| `tech_constraints.技术栈` | `.ai-config/rules/01_tech_stack.mdc` | ✅ `[自动]` | Java 8 + Spring Boot 2.x |
| `tech_constraints.必须兼容` | 从README或规范提取 | ✅ `[自动]` | 支持Chrome/Firefox/Safari |
| 工作量评估的"影响已有功能" | 扫描已有模块，匹配需求关键词 | ⚠️ `[推断]` | "涉及user模块"或"修改order流程" |

### 需要人工确认的字段（推断性）

| 字段 | 数据来源 | 标注方式 | 说明 |
|---|---|---|---|
| `tech_constraints.依赖系统` | 扫描已有模块列表 | ⚠️ `[推断]` | 基于关键词推断，需人工确认是否真的有依赖 |
| `related_requirements` | 搜索历史需求 | ⚠️ `[推断]` | 找到相似需求但不确定是否相关 |
| `stakeholders.影响方` | 扫描已有模块 + 需求关键词 | ⚠️ `[推断]` | "可能影响auth模块"，需确认 |

### 无法自动填写的字段（必须人工）

| 字段 | 原因 | 标注方式 |
|---|---|---|
| `goal` | 需求特定，无法推断 | ❓ `[缺失]` |
| `deadline` | 业务决策，无法推断 | ❓ `[缺失]` |
| `acceptance` | 验收标准，无法推断 | ❓ `[缺失]` |
| `background` | 具体场景背景，无法推断 | ❓ `[缺失]` |

## 标注规则

所有在Step 0A输出或后续Step 1中自动补全的内容，都要标注来源：

### ✅ `[自动]` - 直接从文件读取，确定准确

```markdown
**技术栈**: ✅ `[自动 from .ai-config/rules/01_tech_stack.mdc]`
Java 8 + Spring Boot 2.x + MyBatis-Plus
```

- 直接从规范文件读取，100% 准确
- 不需要人工确认（除非规范本身过期）

### ⚠️ `[推断]` - 基于关键词或逻辑推断，需人工确认

```markdown
**影响已有功能**: ⚠️ `[推断 from src/ 扫描]`
可能涉及 user 模块（因为需求提到"用户权限"）
→ 请确认是否影响 UserController / UserService
```

- 基于代码结构和关键词自动推断
- 概率较高但不 100% 确定
- 用户可以直接确认或修改

### ❓ `[缺失]` - 无法推断，必须人工填写

```markdown
**目标**: ❓ `[缺失，必须由提需人填写]`
需要补充：当前的问题是什么，期望改善到什么程度
```

- 需求特定，无法从项目上下文推断
- 用户必须手工填写
- 不能留空，影响后续流程

## 新项目 vs 旧项目的不同处理

### 新项目（无docs/ 无代码库）

```
识别为新项目后：

【输出建议】
✅ 建议：先初始化项目规范（建立 claude.md 和 .ai-config/）
✅ 建议：选择技术栈（创建 .ai-config/rules/01_tech_stack.mdc）
⚠️ 注意：暂时无历史需求可参考，需求评估时会依赖外部信息

【处理方式】
- 不尝试从代码库加载模块清单（无代码）
- 不搜索历史需求（无历史）
- 工作量评估时，架构维度默认低复杂度（无既有代码影响）
- 如果用户输入的需求是技术架构类，需要特别注意
```

### 旧项目（有docs/ 有代码库）

```
识别为旧项目后：

【自动加载】
✅ 加载技术栈 - 自动填写 tech_constraints
✅ 加载历史需求 - 帮助识别相似功能（伪需求扫描）
✅ 加载已有模块 - 辅助判断"影响已有功能"维度
✅ 加载设计文档 - 参考已有的技术决策

【处理方式】
- 工作量评估时，可以更准确地判断"影响范围"
- 能够识别"这个功能以前做过"的伪需求
- 自动补全的字段更多，澄清问题更少
```

## 异常情况处理

### 情况1：项目混杂（既有新项目特征，也有旧项目特征）

例如：有README但没有src/，有docs/但代码库不完整

```
处理方式：
1. 优先判断"是否有执行过的代码"
2. 如果有，按旧项目处理
3. 如果没有，按新项目处理
4. 在输出中标注"项目状态不清晰"作为警告
```

### 情况2：文档过期（存在docs/但时间戳太旧）

例如：最近的设计文档是2年前的

```
处理方式：
1. 扫描时检查文档时间戳
2. 如果超过6个月未更新，在输出中标注 ⚠️ `[可能过期]`
3. 仍然加载作为参考，但提醒用户verify
```

### 情况3：扫描失败（权限不足、文件损坏等）

```
处理方式：
1. 记录扫描失败的原因
2. 输出中标注 ❌ 
3. 不阻断流程，继续进行Step 0B和Step 1
4. 后续字段无法自动补全（标注为[缺失]）
```
