# Step 0B：输入质量识别规则

## 目标

自动识别用户输入的需求质量等级（T1-T4），并根据质量等级推荐对应的处理策略，以提高需求结构化的效率。

**触发时机**：Step 0A完成后自动执行（无需人工干预）

**执行时间**：2-3秒

**输出**：输入质量等级 + 对应的处理策略

---

## 质量等级判定

### 分级标准

| 等级 | 字数参考 | 信息完整度 | 典型特征 | 示例 |
|---|---|---|---|---|
| **T1**<br/>极简 | <50字 | 极低<br/>仅有目标 | - 一句话<br/>- 无背景<br/>- 无边界<br/>- 无截止日期 | "给用户加个角色权限" |
| **T2**<br/>零散 | 50-200字 | 低<br/>有片段 | - 飞书消息<br/>- 口头记录的文字<br/>- 缺关键字段<br/>- 有背景但无验收 | "昨天产品说要做角色管理，Admin能配置菜单，deadline好像是月底" |
| **T3**<br/>半结构化 | 200-500字 | 中<br/>有框架 | - 有背景/目标/功能<br/>- 缺验收/明确的边界<br/>- 可能有非功能需求<br/>- 但表述零散 | 包含背景和主要功能的飞书消息，但缺少验收标准 |
| **T4**<br/>完整 | 500字+ | 高<br/>近乎完整 | - 完整PRD<br/>- 含验收标准<br/>- 含非功能需求<br/>- 结构清晰 | 完整的产品需求文档 |

### 判定逻辑

```
输入内容
   ↓
统计字数
   ├─ <50 → 判断是否包含明确的目标
   │    ├─ 有 → T1
   │    └─ 无 → 无法处理，要求补充
   │
   ├─ 50-200 → 判断包含的字段数
   │    ├─ 仅1-2个关键字段 → T2
   │    └─ 3个以上字段 → 进一步判断
   │
   ├─ 200-500 → 判断是否有验收标准
   │    ├─ 无 → T3
   │    └─ 有 → 可能是T4
   │
   └─ 500+ → 判断是否有完整的需求结构
        ├─ 有背景/目标/范围/验收 → T4
        └─ 虽长但结构混乱 → T3
```

### 关键字段检查

判定时检查是否包含以下关键字段：

| 字段 | 权重 | 识别方式 |
|---|---|---|
| 目标/目的 | 高 | 寻找"为了...""目标是...""要实现..."等表述 |
| 背景/原因 | 中 | 寻找"因为...""当前...""问题是..."等表述 |
| 功能/范围 | 中 | 寻找"需要...""包括...""功能是..."等表述 |
| 验收/标准 | 高 | 寻找"当...时..."、"完成条件"、"验收"等表述 |
| 截止日期 | 低 | 寻找日期、"月底""下个迭代"等表述 |

---

## 各等级的处理策略

### T1：极简需求 - 引导式问答

**处理理念**：不直接填充模板，而是通过3个关键问题引导用户补充信息。

**执行步骤**：

1. **识别T1** → 输出引导问卷
2. **用户回答** → AI利用回答+项目上下文自动补全其他字段
3. **输出结构化文档**

**问卷模板**：

```markdown
### 🤔 您的需求很简洁！我需要先确认3个关键问题

为了更好地理解这个需求，我需要您回答以下3个问题：

**1️⃣ 这个需求的目标是什么？**
   请用一句话说明：做这个事的目的是什么，期望达成什么效果
   
   💡 例子：
   - "让不同角色的用户看到不同的菜单，提升用户体验"
   - "防止低权限用户越权访问高权限数据"
   - "方便Admin统一管理权限，减少重复配置"
   
   👇 您的回答：

---

**2️⃣ 期望什么时候上线？**
   这是强制截止日期还是目标期限？
   
   💡 例子：
   - "本月底发布（硬deadline）"
   - "下个迭代（目标期限）"
   - "暂无严格要求，按计划来"
   
   👇 您的回答：

---

**3️⃣ 这个需求有什么是不做的，或者有什么限制？**
   请明确说明边界，避免需求蔓延
   
   💡 例子：
   - "不涉及API级权限，只做菜单权限"
   - "不考虑LDAP/AD集成，本期仅支持本地配置"
   - "暂不做权限审计日志，留待下期优化"
   
   👇 您的回答：

---

**回复这3个问题后，我会帮您生成完整的需求文档！**
```

**验收标准**：
- [ ] 3个问题都有明确的答案
- [ ] 答案足以支撑后续的结构化
- [ ] AI可以利用项目上下文补全剩余字段

**处理完成后**：
→ 进入Step 1，利用回答 + 项目上下文生成结构化文档

---

### T2：零散需求 - 主动推断 + 精准澄清

**处理理念**：AI主动推断并生成草稿，标注哪些是推断的（需确认），最多只问3个精准问题。

**执行步骤**：

1. **解析输入** → 提取零散信息
2. **利用上下文** → 基于项目信息主动推断缺失字段
3. **生成草稿** → 标注推断和缺失
4. **精准澄清** → 最多3个关键问题

**输出格式**：

```markdown
### 📝 AI解读（请确认和补充）

基于您的描述和项目上下文，我理解这个需求是：

**一句话总结**：
[用一句话概括需求]

---

### 推断内容（请确认是否正确）

| 字段 | 推断内容 | 确认状态 |
|---|---|---|
| **目标** | ⚠️ `[推断]` 基于您的描述推断为：[推断内容] | [ ] 正确 / [ ] 需修改 |
| **上线日期** | ⚠️ `[推断]` 从"月底"推断为：2024-03-31 | [ ] 正确 / [ ] 需修改 |
| **影响范围** | ⚠️ `[推断]` 可能涉及：user 模块、auth 模块 | [ ] 正确 / [ ] 需修改 |
| **技术栈** | ✅ `[自动]` 从项目规范：Java 8 + Spring Boot 2.x | ✓ 无需确认 |

---

### 关键问题（必须补充）

在上面推断的基础上，还有3个关键问题需要您补充：

**❓ 问题1：[具体澄清问题1]**
   现在状态：[推断/缺失] → [说明为什么问这个问题]
   
   👇 您的回答：

**❓ 问题2：[具体澄清问题2]**
   现在状态：[推断/缺失] → [说明为什么问这个问题]
   
   👇 您的回答：

**❓ 问题3：[具体澄清问题3]**
   现在状态：[推断/缺失] → [说明为什么问这个问题]
   
   👇 您的回答：

---

**回复上面的确认和补充后，我会生成完整的需求文档！**
```

**验收标准**：
- [ ] 推断内容得到确认或修改
- [ ] 3个关键问题都有答案
- [ ] 足以生成完整的结构化文档

**处理完成后**：
→ 进入Step 1，结合推断确认 + 回答生成完整文档

---

### T3：半结构化需求 - 自动补全 + 少量确认

**处理理念**：直接输出完整结构化草稿，标注自动补全部分，最多1-2个确认问题。

**执行步骤**：

1. **提取已有信息** → 从输入中提取所有可用信息
2. **自动补全** → 利用项目上下文补全缺失字段
3. **生成完整文档** → 输出结构化文档草稿
4. **确认关键点** → 最多2个确认问题

**输出格式**：

```markdown
### ✅ 需求文档草稿（已自动补全）

[完整的结构化需求文档，包含所有字段]

---

### 需要您确认的2个关键点

**❓ 确认点1：[具体确认问题]**
   当前状态：[自动补全内容]
   
   [ ] 正确，无需修改
   [ ] 需要修改为：___

**❓ 确认点2：[具体确认问题]**
   当前状态：[自动补全内容]
   
   [ ] 正确，无需修改
   [ ] 需要修改为：___

---

**确认后即可进入评估阶段！**
```

**验收标准**：
- [ ] 文档结构完整
- [ ] 所有自动补全字段都有标注
- [ ] 关键点已确认

**处理完成后**：
→ 直接进入Step 2（工作量评估）

---

### T4：完整需求 - 直接结构化

**处理理念**：直接提取PRD内容，无需澄清问题，自动检查逻辑一致性。

**执行步骤**：

1. **提取内容** → 将PRD内容映射到需求文档字段
2. **自动填补** → 利用项目上下文补全缺失
3. **一致性检查** → 检查goal和acceptance是否对应
4. **输出文档** → 输出完整结构化文档，无需澄清

**输出格式**：

```markdown
### ✅ 需求文档（直接提取自PRD）

[完整的结构化需求文档]

---

### 一致性检查 ✓

✅ 目标和验收标准对应一致  
✅ 需求范围清晰完整  
✅ 无明显矛盾或漏洞  
✅ 无需补充澄清问题

---

**可以直接进入工作量评估！**
```

**验收标准**：
- [ ] 所有字段完整
- [ ] 逻辑一致性通过
- [ ] 无需澄清问题

**处理完成后**：
→ 直接进入Step 2（工作量评估）

## 特殊情况处理

### 情况1：输入无法分类

例如：提供的内容太模糊，既不像T1也不像T2

```
处理方式：
1. 标记为 "无法分类"
2. 输出提示：这个需求信息太模糊，请补充...
3. 建议参考 from_verbal.md 或 from_prd.md 的模板
4. 询问用户：能否再详细说明一下？
```

### 情况2：包含多个独立需求

例如：一次性输入3个不同的功能需求

```
处理方式：
1. 识别出有多个需求
2. 提示用户：检测到您提供了X个不同的需求
3. 建议：请逐个输入，便于我们分别评估和管理
4. 可选：帮用户列出识别的需求清单
```

### 情况3：包含技术方案而非业务需求

例如：输入是"我们应该用Redis做缓存"而不是"需要优化查询性能"

```
处理方式：
1. 识别这是技术方案而非业务需求
2. 提示用户：这更像是一个技术方案，而非需求
3. 反向澄清：您期望通过这个解决什么业务问题？
4. 帮助用户补充：问题 → 目标 → 方案
```

## T1-T4的处理时间预估

| 等级 | 问答往返 | 总耗时 | 澄清次数 |
|---|---|---|---|
| T1 | 3问1答 | 3-5分钟 | 1次（用户回答3问） |
| T2 | 3问1答 | 5-10分钟 | 1次（用户确认推断 + 回答3问） |
| T3 | 2问1答 | 2-3分钟 | 1次（用户确认2个关键点） |
| T4 | 0问 | 1分钟 | 0次（直接生成） |
