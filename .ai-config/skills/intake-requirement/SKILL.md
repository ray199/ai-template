# 技能：需求接入与工作量评估

## 技能描述

将任意形式的原始需求（口头描述、飞书截图、简短消息、会议纪要等）转化为标准结构化需求文档，执行工作量联合评估（需求分析师 + 架构师），根据评估结果确定最终的输出物规范和质量检查方式。

## 触发指令

- `/intake` - 接入并结构化一条新需求，启动工作量评估
- `/intake-check` - 检查已有需求文档的完整性
- `/intake-list` - 列出当前所有需求及其状态
- `/evaluate-workload` - 对已有需求文档重新评估工作量等级

## 完整处理流程（4步）

```
原始需求输入（任意形式）
   ↓
【Step 1】需求结构化（需求分析师）
   - 识别需求来源类型（研发自驱 / 业务方 / 口头）
   - 选择对应模板（from_prd / from_verbal）
   - 填充核心字段，AI自动完整性检查
   - 存在缺失 → 输出澄清问题 → 补充后重新检查
   - 通过检查后，分配 need_id → docs/requirements/backlog/
   ↓
【Step 2】工作量联合评估
   - 需求分析师：业务维度评估（5个维度打分）
   - 架构师代理：技术维度评估（4个维度打分）
   - 参考《workload-evaluation.md》规则，综合判定等级
   - 输出：工作量等级（S / M / L / XL）+ 升级理由（如有）
   ↓
【Step 3】根据工作量等级确定输出物规范
   - S：轻量版（6字段）
   - M：标准版（10字段 + 技术约束）
   - L：完整版（15字段 + 技术思路骨架）
   - XL：完整版 + 迭代拆分计划 + 原型（如有前端）
   - 若当前文档不符合目标等级规格 → 指导补充缺失字段
   ↓
【Step 4】输出物质量检查（自动 or 人工，按工作量分级）
   - S：AI全自动检查（字段完整性 + 格式合规）
   - M：AI初检 + 人工确认业务逻辑
   - L/XL：AI初检（字段完整性）+ 人工终审（内容质量）
   - 检查通过 → 需求进入【阶段2：代码编写】
   - 检查不通过 → 反馈修改意见 → 补充 → 重新检查
```

## 输入格式

### Step 1 输入（需求结构化）

直接粘贴原始内容即可，支持：
- 飞书消息文本
- 口头描述的文字记录
- 会议纪要片段
- 内部技术提案
- 完整PRD文档

```
/intake

[原始需求内容，直接粘贴]
```

### Step 2 输入（工作量评估）

参考 `.ai-config/skills/intake-requirement/workload-evaluation.md` 的评估表单，分别填入：
- 需求分析师的业务维度评分（5个维度）
- 架构师的技术维度评分（4个维度）

示例指令：
```
/evaluate-workload REQ-XXXXXXXX

## 业务维度评估（需求分析师）
- 涉及页面数：3个 → 2分
- 涉及接口数：6个 → 2分
- 数据库变更：新增2张表 → 3分
- 影响已有功能：修改用户模块权限逻辑 → 2分
- 数据迁移：无 → 1分

## 技术维度评估（架构师）
- 架构变更：跨模块权限整合 → 3分
- 引入新技术：无 → 1分
- 性能/并发：现有架构承载 → 1分
- 安全/权限：核心鉴权逻辑变更 → 3分
```

## 输出格式

### Step 1 输出（需求结构化）

若完整性检查通过，输出：

```markdown
# 需求文档：[需求标题]

- **need_id**: REQ-YYYYMMDD-XXX
- **标题**: [已验证，符合20字规范]
- **优先级**: P0 / P1 / P2
- **期望上线日期**: YYYY-MM-DD

## 基本信息
[完整字段内容...]

## 背景与目标
[完整字段内容...]

[其他章节...]

---

## ✅ 完整性检查结果

所有必填字段已验证通过，准备进入【Step 2】工作量联合评估。

指引下一步：
1. 需求分析师完成业务维度评估（参考 workload-evaluation.md）
2. 架构师完成技术维度评估
3. 使用指令 `/evaluate-workload REQ-XXXXXXXX` 提交评估结果
```

若检查不通过，输出澄清问题清单：

```markdown
---
## ⚠️ 需要补充的信息

以下字段必须补充后才能进入工作量评估阶段：

| # | 字段 | 当前状态 | 需要确认 | 重要性 |
|---|---|---|---|---|
| 1 | goal | 未提及 | 目标需要可量化（"将XX从XX提升到XX"） | 必须 |
| 2 | acceptance | 仅1条 | 至少需要2条验收标准 | 必须 |
| 3 | scope.out | 未提及 | 需要明确说明不做什么 | 建议 |

请补充上述字段后，使用指令 `/intake REQ-XXXXXXXX` 重新提交。
```

### Step 2 输出（工作量联合评估）

输出评估报告：

```markdown
# 工作量联合评估报告

- **need_id**: REQ-XXXXXXXX
- **需求标题**: [标题]
- **评估时间**: YYYY-MM-DD

---

## 一、业务维度评估（需求分析师）

| # | 维度 | 评估 | 得分 |
|---|---|---|---|
| 1 | 涉及页面数 | 3个页面（权限列表/编辑/分配） | 2 |
| 2 | 涉及接口数 | 6个接口 | 2 |
| 3 | 数据库变更 | 新增权限表 + 关联表 | 3 |
| 4 | 影响已有功能 | 修改用户模块权限逻辑 | 2 |
| 5 | 数据迁移 | 无 | 1 |
| | **业务总分** | | **10** |

---

## 二、技术维度评估（架构师）

| # | 维度 | 评估 | 得分 |
|---|---|---|---|
| 1 | 架构变更 | 权限模块与用户模块跨模块整合 | 3 |
| 2 | 引入新技术 | 仅使用现有Spring Security | 1 |
| 3 | 性能/并发 | 权限查询不高频，无特殊要求 | 1 |
| 4 | 安全/权限 | 涉及核心鉴权逻辑变更 | 3 |
| | **技术总分** | | **8** |

---

## 三、综合判定

- **基础等级**：按规则表，业务分10 + 技术分8 → M 等级
- **兜底规则检查**：
  - ✓ 安全/权限涉及核心登录鉴权逻辑 → 触发"核心业务流程"规则
  - → **升级为 L**
- **最终等级**：**L（1-2周）**
- **升级理由**：虽基础判定为M，但安全/权限涉及核心鉴权逻辑，按兜底规则升级为L

---

## 四、推荐输出物规范

根据L等级，需求文档需要：
- ✅ 完整版需求文档（15个必填字段）
- ✅ 技术思路骨架（描述架构影响和关键决策点）
- ✅ UI原型（涉及3个新页面）
- ❌ 迭代拆分计划（不需要）
- ❌ 原型（如非前端变更）

**当前需求文档缺失的字段**（需补充）：
- [ ] stakeholders - 干系人列表
- [ ] non_functional - 非功能需求（如性能要求）
- [ ] risks - 已知风险点
- [ ] tech_sketch - 技术思路骨架

确认无误后，进入【Step 3】补充缺失字段，然后 【Step 4】质量检查。

指引下一步：
```
/intake-update REQ-XXXXXXXX

补充以下内容后提交：

## 干系人（stakeholders）
- 提需人：[名字]
- 技术负责人：[名字]
- 影响方：权限管理者、系统管理员

## 非功能需求（non_functional）
- 性能：权限查询 <100ms
- 兼容性：支持已有的Chrome/Safari/Edge
- 安全性：权限变更需审计日志

## 风险点（risks）
- 风险：权限粒度过细导致管理复杂
  影响：中等
  应对：提供权限模板，简化配置
- 风险：与LDAP/AD集成延期
  影响：高
  应对：第一期暂仅支持本地权限

## 技术思路骨架（tech_sketch）
受影响模块：user、permission（新）、auth（改）
初步架构思路：
- 新增 permission 模块，定义权限和角色
- 在 user 模块中添加 role_permission 关联表
- auth 模块集成权限检查，在登录token中携带权限列表
关键决策点：
- 权限粒度：API级别（而非页面级别）
- 权限继承：支持角色继承，不支持权限继承
- 权限缓存：使用Redis缓存用户权限列表，更新后5秒内生效
```

### 无法解析或不符合规范时

```markdown
## ❌ 无法完成结构化

原始内容信息量不足，无法识别以下核心要素：
- [ ] 需求目标不明确（无法回答"为什么要做"）
- [ ] 功能边界未定义（不清楚做什么/不做什么）
- [ ] 验收标准不可验证（"做好一点"等模糊表述）
- [ ] 其他：[说明]

建议：请参考模板 `.ai-config/skills/intake-requirement/templates/from_verbal.md` 补充信息后重新提交。
```

## Step 1 完整性检查规则（结构化阶段）

### S1-基础必填字段（所有模板都需要）

| 字段 | 检查规则 | 阻断 |
|---|---|---|
| `need_id` | 自动分配（格式REQ-YYYYMMDD-XXX） | ✅ |
| `title` | 不超过20字，包含动词+对象 | ✅ |
| `priority` | P0 / P1 / P2，需说明定级理由 | ✅ |
| `deadline` | 具体日期或明确的里程碑（不接受"尽快"） | ✅ |
| `goal` | 必须可量化（有数字或可验证的状态），至少1条 | ✅ |
| `acceptance` | 至少1条验收标准，格式："当[条件]时，系统应[行为]" | ✅ |

> 这6个字段是【Step 1】的必须项，缺失任何一个都要求补充

### S1-辅助字段（基于来源类型）

| 字段 | from_prd | from_verbal | 检查规则 |
|---|---|---|---|
| `提需人` / `接收人` | 必填 | 必填 | 非空 |
| `background` | 必填 | 必填 | 至少说明业务场景和当前问题 |
| `scope.in` / `scope.out` | 必填 | 必填 | 同时包含包含和不包含两部分 |
| `tech_constraints` | 可选 | 可选 | 如有则需列举 |
| `feasibility` | 可选 | 必填 | 初步判断：已评估-可行 / 已评估-存在风险 / 待评估 |

> 这些字段如缺失，AI会输出澄清问题，可选字段空白则标记为可选补充

---

## Step 2 工作量评估的输出物质量规则

不在Step 1阶段检查，而是在【Step 2】工作量联合评估时检查：

### 需求分析师评估质量检查

| 检查项 | 合规条件 |
|---|---|
| 业务维度5项均已评分 | 每项都在1-3分范围内，且已说明关键影响 |
| 得分在5-15范围内 | 5项分数之和应在此范围 |
| 关键影响说明充分 | 3分项必须解释为什么是高复杂度 |

### 架构师评估质量检查

| 检查项 | 合规条件 |
|---|---|
| 技术维度4项均已评分 | 每项都在1-3分范围内，且已说明关键风险 |
| 得分在4-12范围内 | 4项分数之和应在此范围 |
| 关键风险说明充分 | 3分项必须说明技术风险是什么 |

---

## Step 3/4 输出物完整性检查规则（按工作量等级）

### S等级（轻量版）

**必填字段（6个）**：need_id + title + priority + deadline + goal (1条) + acceptance (1条)

**检查方式**：AI全自动

| 检查项 | 合规条件 |
|---|---|
| 字段完整性 | 6个字段全部非空 |
| 格式合规 | priority 为 P0/P1/P2，deadline 为日期，goal 可量化，acceptance 有"当...时，系统应..." |

**通过标准**：所有项✅ → 直接进入【阶段2代码编写】

---

### M等级（标准版）

**必填字段（10个）**：S的全部 + background + scope.in + scope.out + tech_constraints + acceptance (≥2条) + goal升级为(1-3条)

**检查方式**：AI初检 + 人工确认

| 检查项 | AI检查 | 人工确认 |
|---|---|---|
| 字段完整性 | ✅ 10个字段非空 | ✅ 逻辑一致（目标与验收标准对应） |
| 格式合规 | ✅ goal可量化，acceptance格式正确 | ✅ 业务逻辑准确，目标合理可达 |
| 范围清晰 | ✅ 包含和不包含都有 | ✅ 边界明确，不存在歧义 |

**通过标准**：AI✅ + 人工✅ → 进入【阶段2代码编写】

---

### L等级（完整版）

**必填字段（15个）**：M的全部 + stakeholders + non_functional + risks + tech_sketch + acceptance (≥3条) + 可选：prototype_link

**检查方式**：AI初检 + 人工终审（必须）

| 检查项 | AI初检 | 人工终审 |
|---|---|---|
| 字段完整性 | ✅ 15个核心字段非空 | ✅ stakeholders清单完整（含提需人/负责人/影响方） |
| 技术思路质量 | ✅ tech_sketch存在 | ✅ 架构思路与现有代码库一致，关键决策点明确 |
| 风险识别 | ✅ risks至少有1项 | ✅ 风险点合理，应对方案切实可行 |
| 非功能需求 | ✅ non_functional非空 | ✅ 性能/安全/兼容性要求可测试 |
| 原型匹配 | ✅ 涉及前端时prototype_link非空 | ✅ 原型与需求描述一致 |

**通过标准**：AI✅ + 人工✅ → 进入【阶段2代码编写】

---

### XL等级（超大型，需拆分）

**必填字段**：L的全部 + iteration_plan + milestones + prototype_link(前端必须)

**检查方式**：AI初检 + 人工终审 + 团队评审（三级）

| 检查项 | AI初检 | 人工终审 | 团队评审 |
|---|---|---|---|
| 拆分合理性 | ✅ iteration_plan中每个子需求独立交付 | ✅ 拆分粒度合适，避免过细或过粗 | ✅ 整体方向正确，不存在遗漏 |
| 里程碑明确 | ✅ milestones有时间节点 | ✅ 各阶段目标清晰 | ✅ 资源分配可行，不超过团队能力 |
| 依赖梳理 | ✅ 子需求间依赖关系明确 | ✅ 实现顺序合理 | ✅ 并行/串行安排已确认 |

**通过标准**：AI✅ + 人工✅ + 团队✅ → 拆分后的每个子需求各自走【阶段1-4】完整流程

## AI自动检查规则（用于Step 1和Step 3/4）

### 字段存在性检查
```
FOR each 必填字段 in 当前等级 DO
  IF 字段值为空或未提及 THEN
    输出澄清提示 + 关键问题说明
  ELSE
    继续检查格式
  END IF
END FOR
```

### 格式合规性检查
```
IF title长度 > 20字 OR 不含动词 THEN 标红
IF priority NOT IN [P0, P1, P2] THEN 标红
IF deadline 不是日期格式且不是明确里程碑 THEN 标红
IF goal 不含数字且不可验证 THEN 标红
IF acceptance 不符合"当...时，系统应..."格式 THEN 标红
...
```

### 逻辑一致性检查（M级及以上）
```
IF goal和acceptance不对应 THEN 警告
IF scope.in和scope.out有重叠 THEN 错误
IF tech_sketch中提到的模块与影响面评估不一致 THEN 警告
...
```
