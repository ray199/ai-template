**核心功能**
| 功能 | 说明 |
|---|---|
| 阶段划分 | 需求接入 → 工作量评估 → 输出物规范 → 质量检查 |
| 输出物规范 | 定义每个阶段应该产出什么文档 |
| 工作量分级 | S/M/L/XL四个等级，对应不同规格 |
| 确认指令 | /intake, /evaluate, /design, /code等 |

**依赖关系**：独立，是所有其他模块的基础

### 3.3 阶段0A/0B: 需求接入 (.ai-config/skills/intake-requirement/)

**职责**：将任意形式的需求转化为结构化需求文档

**包含的技能**
| 文件 | 用途 | 输入 | 输出 |
|---|---|---|---|
| **SKILL.md** | 触发和流程定义 | /intake指令 | 完整流程说明 |
| **project-context.md** | Step 0A: 自动扫描项目 | 项目目录 | 项目上下文摘要 |
| **input-quality.md** | Step 0B: 质量分级 | 原始需求 | T1/T2/T3/T4等级 |
| **workload-evaluation.md** | 工作量评估规则 | 结构化需求 | S/M/L/XL等级 |
| **templates/** | 需求模板 | 填充内容 | REQ文档 |

**流程**
```
原始需求 (任意形式)
   ↓
Step 0A: 项目上下文感知
   ├─ 扫描项目，识别新/旧项目
   ├─ 加载可用资源
   └─ 输出: 项目上下文摘要
   ↓
Step 0B: 输入质量识别
   ├─ 判断T1/T2/T3/T4
   └─ 输出: 推荐处理策略
   ↓
Step 1: 需求结构化
   ├─ 根据质量采用对应策略
   ├─ 自动填补已知字段
   └─ 输出: 结构化需求文档 (docs/requirements/backlog/)
   ↓
Step 2: 工作量评估
   ├─ 业务维度打分
   ├─ 技术维度打分
   └─ 输出: S/M/L/XL等级
```

**依赖关系**：需要project-context.md中的技术栈信息

## 四、模块间通信和协作

### 跨模块调用规则

#### 需求接入 → 工作量评估
```
需求接入完成 (REQ-XXXXX.md生成)
   ↓
触发工作量评估 (/evaluate-workload REQ-XXXXX)
   ├─ 读取REQ文档的: 目标、功能点、验收标准
   ├─ 参考TECH-STACK.md中的技术约束
   └─ 输出工作量等级 (S/M/L/XL)
```

#### 工作量评估 → 编码准备
```
工作量等级确定后 (S/M/L/XL)
   ↓
根据等级执行Pre-Coding检查
   ├─ S等级: 跳过检查，直接编码
   ├─ M等级: 轻量检查，确认基础信息
   ├─ L等级: 完整检查，包括设计评审
   └─ XL等级: 超完整检查，需要拆分
```

#### 编码阶段 → 项目扫描
```
新需求完成编码后
   ↓
更新项目快照 (/scan)
   ├─ 更新PHASE3-PROJECT-SCANNING.md的"最近活动"
   ├─ 更新MODULES.md的模块依赖（如有新模块）
   ├─ 更新TECH-STACK.md的技术债或已知问题
   └─ Git提交: "chore: update project snapshot"
```

4. 交付和更新
   ├─ 合并到主分支
   ├─ 更新MODULES.md

## 七、模块交互中的常见问题

### Q: 如何避免模块间的循环依赖？
**A**: 遵循分层原则：
```
【分层】
┌─────────────────────┐
│ Controller层 (API)  │ 只依赖 Service层
├─────────────────────┤
│ Service层 (业务)    │ 只依赖 Mapper/Repository层
├─────────────────────┤
│ Mapper/Repo层       │ 只依赖 Entity和DB
├─────────────────────┤
│ Entity层 (数据)     │ 不依赖任何层
└─────────────────────┘

【跨模块通信】
userModule ──┐
             ├─→ 调用orderModule.OrderService
orderModule ──┘
             （不要反过来调用）
```

### Q: 如何处理模块间的参数传递？
**A**: 使用DTO而不是Entity：
```java
// ❌ 错误：暴露内部Entity
userModule.export User getUser(long id);  // User是Entity

// ✅ 正确：使用DTO
userModule.export UserDTO getUserDTO(long id);  // UserDTO是传输对象
```

### Q: 模块间能否共享数据库表？
**A**: 遵循以下原则：
```
【表的所有权】
每个表只属于一个模块，该模块拥有所有权

【跨模块数据访问】
- 通过API接口访问（推荐）
  userModule暴露: UserService.getUserById(id)
  
- 不允许直接查询其他模块的表
  ❌ orderModule.selectFromUserTable() 
  
- 允许引用外键，但不允许JOIN查询
  ✅ order表中有user_id字段
  ❌ order表中不能直接关联user表做JOIN
```

## 九、常见模块设计错误

| 错误 | 症状 | 解决方案 |
|---|---|---|
| **循环依赖** | 编译失败，无法初始化 | 重新分层，引入中间层 |
| **过度耦合** | 一个模块改动影响所有模块 | 使用接口，减少直接依赖 |
| **责职不清** | 多个模块做同样的事 | 明确模块的单一职责 |
| **过大模块** | 单个模块>10000行代码 | 拆分为多个专业模块 |
| **隐式依赖** | 需要特定初始化顺序 | 使用显式注入，避免静态方法 |
| **数据重复** | 多个模块存储相同数据 | 确定所有权，通过API访问 |
