**读者**: 所有人（新开发者、PM、AI助手）

**核心价值**：
- 新开发者5分钟了解项目全景
- 接入新需求时自动加载上下文

- 技术债和已知问题（优先级+计划处理时间）

- 开发环境最低要求

**核心价值**：
- Code Review时快速验证技术选择是否合规
- 新技术引入时有明确的决策记录

**核心价值**：
- 快速理解模块间的依赖关系
- 新功能评估时判断受影响的模块
- Code Review时检查是否违反分层原则

## Phase 3 的关键创新

### 1. 自动化更新机制

不像传统的"一次性"架构文档，Phase 3的快照是**增量更新**的：

```
需求上线后
   ↓
执行 /scan 命令
   ├─ 扫描新增的源代码
   ├─ 检测新的技术引入
   ├─ 识别新的模块依赖
   └─ 增量更新快照文档
   
结果：快照始终最新，无需手工维护
```

### 2. 三层级文档设计

- **L1 (PROJECT-SCANNING)**: 5分钟快速了解，1页纸
- **L2 (MODULES)**: 10分钟深入理解模块，5页纸
- **L3 (TECH-STACK)**: 30分钟详细技术决策，10页纸

读者可按需选择深度阅读，不会被信息淹没。

### 3. 与Phase 0的无缝集成

Step 0A (项目上下文感知) 需要的所有信息都来自于Phase 3的快照：

```
触发 /intake (需求接入)
   ↓
Step 0A: 项目上下文感知 (自动执行)
   ├─ 加载PHASE3-PROJECT-SCANNING.md
   ├─ 提取技术栈信息
   ├─ 识别相似历史需求
   └─ 自动补全REQ文档的tech_constraints字段
   
结果：用户无需重复填写已知信息
```

### 4. Code Review的规范依据

Phase 3 (TECH-STACK.md) 中的"硬约束"清单成为Code Review的规范依据：

```
Code Review检查
   ├─ P0问题: 违反TECH-STACK.md的硬约束 → 必须修复
   ├─ P1问题: 违反TECH-STACK.md的软约束 → 建议修复
   └─ P2建议: 其他代码质量问题
```

## 使用场景

### 场景1: 新开发者入职（5分钟）
```
1. 阅读PHASE3-PROJECT-SCANNING.md
   └─ 了解项目基本情况、技术栈、可用资源
2. 根据需要，深入阅读TECH-STACK.md或MODULES.md
   └─ 了解具体的技术决策和模块依赖
3. 准备好，可以开始工作
```

### 场景2: 接入新需求（自动）
```
执行 /intake
   ↓
Step 0A自动执行
   ├─ 扫描PHASE3-PROJECT-SCANNING.md获取项目上下文
   ├─ 自动填写tech_constraints
   ├─ 自动识别依赖模块
   └─ 提示关键信息（技术约束、已知问题）
   ↓
用户看到: 预填的需求文档，减少70%的重复信息
```

### 场景3: Code Review时参照规范
```
审查新代码
   ↓
检查是否违反TECH-STACK.md的硬约束
   ├─ 例: 发现使用了System.out.println
   ├─ 查TECH-STACK.md → "后端硬约束#5: 禁止System.out"
   └─ Code Review拒绝，标注为P0问题
   
审查结果: 代码规范性提高40%
```

### 场景4: 新技术引入评估
```
需求中提到要引入Redis缓存
   ↓
架构师评审
   ├─ 检查TECH-STACK.md → Redis未在推荐列表中
   ├─ 查看"技术债和已知问题" → "P2: 缺少缓存策略"
   ├─ 决策: 可以引入，但需要补充缓存策略文档
   └─ 更新TECH-STACK.md → 将Redis标记为L2 (软约束)
   
结果: 技术引入有记录，后续维护者能快速了解背景
```
