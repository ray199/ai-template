# 编码准备检查表

## 目标

在编码开始前，确保所有准备工作都已完成，减少编码中的返工和问题。

---

## Step 1：需求阶段输出物检查

### 需求文档检查

```
【必填项】
- [ ] 需求ID已分配（REQ-XXXXXXXX）
- [ ] 需求标题清晰完整
- [ ] 需求目标(goal)已定义
- [ ] 验收标准(acceptance)≥2条，且清晰可测
- [ ] 优先级已确定
- [ ] 截止日期已确定

【规范项】
- [ ] 背景和原因清晰
- [ ] 功能范围明确（scope.in/out）
- [ ] 技术约束(tech_constraints)已标注
- [ ] 影响范围已评估
- [ ] 相关需求已列出

【原型项】（如有）
- [ ] 原型文件存在
- [ ] 原型与需求对应清晰
- [ ] 原型和需求之间无矛盾
```

### 工作量评估检查

```
- [ ] 工作量等级已确定（S/M/L/XL）

- [ ] 推荐输出物规范清晰

```

## 常见问题排查

### 问题：需求文档有歧义

**检查清单：**
- 原型中是否有说明？
- 是否与需求提出方再次确认？
- 是否在Code Review时标注为P1问题？

### 问题：技术选型有疑虑

**检查清单：**
- 是否查看了项目现有技术栈？
- 是否考虑了技术债？
- 是否与架构师讨论过？

### 问题：API接口设计不够清晰

**检查清单：**
- 是否生成了Swagger文档？
- 是否提供了请求/响应示例？
- 是否前后端都认可？

### 问题：数据库字段名争议

**检查清单：**
- 是否遵循了项目现有的命名规范？
- 是否与DBA讨论过性能影响？
- 是否所有团队成员都认可？
