# AI编程工作流完整指南

## 📋 项目概览

这是一套完整的AI编程工作流系统，集成4个核心阶段（需求接入→设计→编码→交付）和3个支撑阶段（项目扫描→测试→审查），支持从一句话需求到生产发布的完整闭环。

**核心创新**: 根据需求复杂度（S/M/L/XL）自动调整输出物规范和检查严度，避免"一刀切"的低效方案。

---

```
┌─────────────────────────────────────────────────────────────────────┐
│                  AI 编程完整工作流（6阶段）                          │
└─────────────────────────────────────────────────────────────────────┘

【阶段0】项目扫描与上下文准备 (Phase 3)

   ├─ Step 1: 需求结构化 (字段补全 + 自动检查)
   ├─ Step 2: 工作量联合评估 (业务+技术双维评分)
   ├─ Step 3: 根据等级补充输出物
   └─ Step 4: 质量检查 (按等级分级: AI全自动/人工确认/人工终审)

   ↓
【阶段2】技术设计 (M/L/XL必需, S可跳过)
   ├─ 架构影响分析 (受影响模块、技术决策)
   ├─ 数据库设计 (DDL + 迁移 + 回滚方案)
   ├─ 接口设计 (API清单、请求/响应、错误码)
   └─ 关键实现路径 (核心流程、并发处理、风险点)
   ↓ 输出: 设计文档 + 评审确认
   ↓
【阶段3】编码实现
   ├─ Entity/DTO/VO生成
   ├─ Mapper + Service + Controller生成
   ├─ DB迁移脚本 + 测试骨架
   └─ 代码规范自检 (遵循02_code_style + 03_security规范)
   ↓ 输出: 源代码 + 测试 + 编码完成报告
   ↓
【阶段4】测试与审查
   ├─ 单元测试执行 (覆盖率 ≥80%)
   ├─ 集成测试执行 (关键流程验证)
   ├─ 代码审查 (5个维度: 质量/架构/安全/可维护/业务完整)
   └─ 问题修复与重新审查
   ↓ 输出: 测试报告 + 审查报告
   ↓
【阶段5】交付与上线
   ├─ 交付检查 (代码质量、文档、部署脚本)
   ├─ 部署前验证 (灾备方案确认)
   ├─ 执行上线 (DB迁移 + 应用部署 + 验证)
   └─ 回滚方案验证
   ↓ 输出: 发布总结 + 上线文档
   ↓
✅ 交付完成
```

## 📊 工作量分级 (S/M/L/XL) 决定一切

### 分级规则

**工作量计算公式**：
- 业务维度得分 (5项): 涉及页面数 + 接口数 + DB变更 + 已有功能影响 + 数据迁移
- 技术维度得分 (4项): 架构变更 + 新技术引入 + 性能/并发 + 安全/权限

| 等级 | 业务分 | 技术分 | 特征 | 工期 |
|---|---|---|---|---|
| **S** | ≤7 | ≤6 | 简单功能，一两页面，新增字段 | 1-2天 |
| **M** | 8-10 | 7-8 | 标准需求，2-4页面，新表，跨模块改动 | 3-5天 |
| **L** | 11-13 | 9-10 | 复杂需求，5+页面，有风险，核心逻辑变更 | 1-2周 |
| **XL** | ≥14 | ≥11 | 超大型，多个3分项，需拆分成子需求 | 需拆分 |

### 各等级的输出物规范

| 等级 | 需求文档 | 技术设计 | 原型 | 审查方式 | 输出物数量 |
|---|---|---|---|---|---|
| **S** | 6字段 | ❌ 无 | ❌ 无 | AI自动 | 3-4个 |
| **M** | 10字段 | ⚠️ 轻量 | ⚠️ 如需 | AI初检+人工确认 | 5-6个 |
| **L** | 15字段 | ✅ 完整 | ✅ HTML原型 | AI初检+人工终审 | 7-8个 |
| **XL** | 15字段+拆分 | ✅ 完整 | ✅ HTML+Figma | AI初检+人工终审+团队评审 | 10+个 |

## 📥 阶段1：需求接入工作流详解

### 触发指令
```bash
/intake                    # 启动需求接入 (自动执行Step 0A/0B)
/evaluate-workload REQ-XXX # 启动工作量评估
/intake-update REQ-XXX     # 补充缺失字段
```

### 5步完整流程

#### Step 0A: 项目上下文自动感知 (5-10秒)
**自动执行，无需用户操作**

系统扫描并输出：
- 项目基本信息 (新/旧项目、技术栈)
- 可用资源清单 (规范、历史需求、设计文档、模块)
- 自动补全的字段 (标注来源)

**关键输出**：
```
✅ 自动加载: 技术栈 → Java 8 + Spring Boot 2.7 + MyBatis Plus
⚠️ 推断: 影响模块 → user, auth (基于代码分析)
❓ 缺失: 期望上线日期 (等待用户输入)
```

#### Step 0B: 输入质量自动识别 (2-3秒)
**自动执行，无需用户操作**

识别需求等级并推荐策略：

| 输入等级 | 特征 | 推荐策略 | 用户工作量 |
|---|---|---|---|
| **T1** (一句话) | "给用户加角色权限" | 3个引导问题 | 3-5分钟 |
| **T2** (零散) | "产品要做XX，deadline月底，可能影响YY模块" | 推断+3个澄清 | 5-10分钟 |
| **T3** (半结构化) | PRD的核心部分已有，缺补充 | 自动补全+2个确认 | 2-3分钟 |
| **T4** (完整PRD) | 完整的PRD或结构化需求 | 直接结构化 | 0分钟 |

## 🎯 最佳实践

### 需求接入阶段
1. **一句话也能接入** - 不要等到需求完美再提，T1也能处理
2. **信任自动补全** - 系统会自动识别项目上下文，填写已知信息
3. **专注澄清缺失项** - 只需要回答系统提出的问题，无需重复填写

### 工作量评估阶段
1. **双维度评分很关键** - 业务复杂度 + 技术风险都要考虑
2. **兜底规则要重视** - 单个核心逻辑 3分项也要升级为L
3. **及早识别XL需求** - 越早拆分，后续执行越顺畅

### 设计和编码阶段
1. **S等级可以跳过设计** - 简单功能，直接编码最高效
2. **L等级必须有完整设计** - 大型功能，前期设计能省后期返工
3. **代码规范来自TECH-STACK** - Code Review时直接对标 Phase 3 的硬约束

### 交付上线
1. **事前备份验证** - 关键表一定要备份脚本
2. **分阶段上线** - XL需求采用灰度发布
3. **回滚预案要测试过** - 不要到出问题才想起怎么回滚
