# AI编程规范模板

一套完整的AI编程工作流系统，集成需求接入→设计→编码→交付的6个核心阶段，支持从一句话需求到生产发布的完整闭环。

**核心创新**: 根据需求复杂度（S/M/L/XL）自动调整输出物规范和检查严度，避免"一刀切"的低效方案。

## 项目介绍

这是一个专为AI编程设计的规范驱动开发工具模板，结合OpenSpec和Superpowers方法论，通过轻量级的配置和标准化目录结构，让规范驱动流程可快速落地。

**已完成工作**:
- ✅ Phase 1: 需求接入与工作量评估（5步流程）
- ✅ Phase 2: 技术设计（完整模板）
- ✅ Phase 3: 项目自动扫描与快照（项目上下文+技术决策+模块依赖）
- ✅ Phase 4: 编码实现（代码生成+规范自检）
- ✅ Phase 5: 测试与审查（测试模板+审查清单）
- ✅ Phase 6: 交付上线（部署计划+回滚方案）

## 核心优势

- **支持主流AI工具**：兼容Claude Code、Cursor等主流AI工具
- **无需复杂配置**：新手开发者也能快速上手，T1一句话即可接入
- **完整开发闭环**：从需求接入到生产部署的完整流程
- **可复用技能**：支持动态加载和组合技能
- **智能工作量评估**：根据复杂度自动调整输出物规范
- **自动化上下文**：项目扫描和上下文自动加载，减少重复信息

## 📊 工作量等级速查表

| 等级 | 特征 | 工期 | 输出物 | 审查方式 |
|---|---|---|---|---|
| **S** | 简单（<3页面、新字段） | 1-2天 | 6项 | AI自动 |
| **M** | 标准（3-5页面、新表） | 3-5天 | 6项 | AI+人工 |

**快速判断**: 一个功能涉及的页面数和数据库变更规模最能判断工作量。

## 🎯 核心功能

### 【阶段1】需求接入与工作量评估
- **自动项目上下文感知**: 扫描项目、加载技术栈、识别相似需求
- **智能输入质量识别**: T1(一句话)→T4(完整PRD)，自动采用对应处理策略
- **需求自动补全**: 利用项目上下文自动填写已知字段，标注来源
- **工作量双维评估**: 业务(5项) + 技术(4项)联合打分，确定S/M/L/XL等级
- **按等级补充字段**: 自动提示缺失字段，根据等级要求补充
- **分级质量检查**: S(自动)、M(初检+确认)、L(初检+终审)、XL(三级评审)

### 【阶段2】技术设计
- **架构影响分析**: 识别受影响模块、技术决策、变更范围
- **完整数据库设计**: DDL + 迁移脚本 + 回滚方案
- **详细接口设计**: API清单、请求/响应、错误码、权限说明
- **关键实现路径**: 核心流程、并发处理、性能优化、风险应对

### 【阶段3】编码实现
- **代码自动生成**: Entity/DTO/VO/Mapper/Service/Controller
- **规范自检**: 遵循代码风格、安全、性能、注释规范
- **DB脚本生成**: 迁移脚本 + 回滚脚本
- **测试骨架**: 单元测试框架和用例模板

### 【阶段4】测试与审查
- **单元测试**: 自动执行，统计覆盖率(目标≥80%)
- **集成测试**: 核心流程端到端验证
- **代码审查**: 5个维度(质量/架构/安全/可维护/完整性)
- **问题跟踪**: 记录问题等级、修复情况、重新审查

### 【阶段5】交付上线
- **交付清单**: 代码质量、文档、部署脚本、灾备方案
- **部署计划**: 分阶段上线、灰度发布、流量切割
- **回滚预案**: 完整的回滚脚本和验证步骤
- **上线文档**: 变更摘要、风险评估、联系方式

### 【Phase 3】项目扫描与快照（支撑阶段）
- **PHASE3-PROJECT-SCANNING.md**: 5分钟快速了解项目全景(1页纸)
- **TECH-STACK.md**: 技术决策和约束(10页)，Code Review的规范依据
- **MODULES.md**: 模块清单、依赖关系、交互规则(5页)

```bash
# 【阶段1】需求接入
/intake                          # 启动需求接入 (自动化5步流程)
/intake-update REQ-20260406      # 补充缺失字段

# 【阶段2】技术设计 (M/L/XL必需)
/design REQ-20260406             # 生成设计文档
/design-confirm REQ-20260406     # 确认设计，进入编码

# 【阶段3】编码实现
/code REQ-20260406               # 生成源代码 (Entity/Mapper/Service/Controller)
/code-confirm REQ-20260406       # 确认代码完整，进入测试

# 【阶段4】测试与审查
/test REQ-20260406               # 执行单元+集成测试
/test-confirm REQ-20260406       # 确认测试通过
/review REQ-20260406             # 启动代码审查
/review-fix REQ-20260406         # 提交修复后重新审查

# 【阶段5】交付上线
/deliver REQ-20260406            # 生成交付文档和部署计划
```

## 💡 最佳实践

### 需求接入阶段
- **一句话就能启动** - 不必等待完美需求，T1质量也能处理，系统会引导补充
- **信任自动补全** - 系统自动识别项目上下文，无需重复填写已知信息
- **专注澄清缺失** - 只需回答系统提出的问题，剩余字段自动推断
- **核心业务升级** - 简单功能涉及核心逻辑(支付/登录/订单)要升级为L等级

### 工作量评估阶段
- **双维度打分很关键** - 业务复杂度 + 技术风险都要认真评估
- **兜底规则要重视** - 单个3分项(核心业务)也要升级为L，避免低估
- **及早识别XL需求** - 越早拆分为L/M子需求，后续执行越顺畅

### 设计和编码阶段
- **S等级跳过设计** - 简单功能(加字段)直接编码最高效，无需设计文档
- **L等级必须设计** - 大型功能前期充分设计能省30%的后期返工
- **代码规范参考TECH-STACK** - Code Review时直接对标 `.ai-config/rules/` 的硬约束

### 交付上线阶段
- **灾备验证不能少** - 关键表必须有备份脚本并实际测试过
- **分阶段上线** - XL需求采用灰度发布(10% → 50% → 100%)而非全量发布
- **回滚预案要测试** - 不要到出问题才想起怎么回滚，事前演练很重要
